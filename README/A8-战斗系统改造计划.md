# A级文档：战斗系统改造计划

## 1. 目标

把当前“轮流行动”的战斗流程，改造成：

- 轻量 `AV 时间轴系统`
- `终结技插入系统`
- `节奏收益` 可被玩家直接理解的战斗模型

这份文档回答的是“怎么改”，不是“为什么改”。

## 1.1 未来开发目标：共享战斗维度必须有落点

在正式编码前，先冻结以下共享战斗维度作为未来开发目标：

- `速度`
  要从“先手修饰”升级为“时间轴主属性”
- `防御`
  要从“纯减伤”升级为“反击、受击充能、硬吃换轴”的挂点
- `生命 / 已损生命`
  要从“容错值”升级为“残血收益、失血收益、过量治疗收益”的挂点
- `法力`
  要从“支付成本”升级为“过载、循环、资源换节奏”的挂点
- `攻击`
  要从“伤害倍率”升级为“重击、破防、处决、压制”的挂点
- `终结能量`
  要成为独立于法力和职业资源之外的共享战斗资源
- `韧性 / 失衡 / 架势`
  要成为打断、处决、压制和反压制的统一接口

后续的职业、遗物和装备流派，优先围绕这些共享战斗维度生长。

## 2. 当前实现的根问题

当前 [combat.js](/F:/HTML2DRPG/combat.js) 的核心仍然建立在：

- `playerTurn`
- `runEnemyTurn()`
- 玩家行动后固定进入敌方行动

这意味着：

- 速度只影响先手和少量附加收益
- 行动顺序不可操控
- 技能只以伤害 / 资源成本组织
- 终结技无法以“插入行为”存在

因此，这次改造必须重写的是 `战斗推进模型`，不是只补几个字段。

## 2.1 这轮技术改造要为哪些未来流派预留空间

这轮不做职业平衡，但技术上必须提前给以下流派留好挂点：

- `快轴流`
  高频行动、短延迟、抢轴、插终结技
- `慢轴重击流`
  大延迟换高收益、蓄力、处决
- `打断压制流`
  围绕韧性 / 失衡推进战斗
- `受击反打流`
  通过防御、护盾或受击资源反推输出
- `低血爆发流`
  围绕已损生命和风险收益展开
- `资源过载流`
  围绕法力或职业资源换节奏展开
- `终结技循环流`
  围绕终结能量的获取、保留和插入时机展开

如果技术底座不能自然承接这些流派，说明战斗系统的改造仍然不够彻底。

## 3. 改造总路线

建议按 5 个阶段推进。

### R1. 抽离行动序列核心

新增独立的时间轴模块，例如：

- `combat-timeline.js`

负责：

- 维护参战单位列表
- 维护每个单位的 `currentAv`
- 计算谁先获得行动权
- 推进到下一个行动节点
- 输出时间轴快照

这一层不能直接依赖 DOM。

### R2. 技能协议升级

把技能从“倍率 + cost”升级为：

- `baseDelay`
- `speedScale`
- `advanceSelf`
- `delayTarget`
- `ultimateChargeGain`
- `ultimateChargeCost`
- `canInsertUltimate`

这样技能本身才能参与时间轴。

### R3. 终结技插入协议

新增终结技运行协议：

- 普通技能和终结技分流
- 终结技拥有独立资源槽
- 终结技可在指定窗口插入
- 插入后触发单独事件
- 插入结果会回写时间轴

### R4. 时间轴 UI 与战斗菜单升级

最低新增：

- 时间轴预览条
- 当前行动者高亮
- 技能延迟与插队说明
- 终结技可插入入口
- 终结技资源显示

### R5. 验证链路与烟雾测试升级

新增可运行验证：

- 时间轴初始化正确
- 速度差会改变行动顺序
- 技能延迟会改变下一次行动时机
- 终结技插入不会破坏普通回合推进
- 时间轴快照可以被 UI 正常读取

## 4. 建议新增的数据结构

## 4.1 CombatUnitRuntime

建议每个战斗单位运行时都具备：

- `unitId`
- `side`
- `hp`
- `maxHp`
- `speed`
- `statuses`
- `currentAv`
- `baseAv`
- `ultimateCharge`
- `ultimateMax`
- `canAct`
- `tags`

## 4.2 CombatTimelineState

建议新增：

- `actors`
- `currentActorId`
- `roundIndex`
- `queuePreview`
- `pendingInsertWindow`
- `lastResolvedAction`

## 4.3 CombatActionSpec

建议所有战斗行动统一走：

- `actionId`
- `actionType`
  可取：`skill` / `ultimate` / `flee` / `wait`
- `sourceUnitId`
- `targetUnitId`
- `baseDelay`
- `advanceSelf`
- `delayTarget`
- `resourceChanges`
- `effects`

## 4.4 UltimateInsertWindow

建议新增独立结构：

- `open`
- `sourceUnitId`
- `allowedActionIds`
- `reason`
- `expiresAt`

用来告诉 UI：

- 现在能不能插终结技
- 谁能插
- 可以插哪些终结技
- 这个窗口何时关闭

## 5. 现有文件的改造建议

## 5.1 [combat.js](/F:/HTML2DRPG/combat.js)

这是主改造文件。  
建议改成“战斗协调器”，而不是继续承担完整战斗规则。

应保留：

- `createCombatController`
- 战斗外层输入输出
- 与 `CombatIO` 的通信

应移出：

- 先手判定
- 回合推进
- 行动顺序计算
- 插入逻辑

## 5.2 新增 `combat-timeline.js`

负责：

- 时间轴初始化
- AV 推进
- 行动顺序决议
- 插入窗口管理
- 时间轴快照输出

## 5.3 新增 `combat-actions.js`

负责：

- 根据技能配置生成 `CombatActionSpec`
- 区分普通技能与终结技
- 统一描述延迟、提前、延后、资源变更

## 5.4 新增 `combat-effects.js`

负责：

- 伤害
- 恢复
- 状态附加
- 资源变动
- 时间轴变动

这一层只吃动作规格和单位运行态，不直接碰 UI。

## 5.5 [combat-io.js](/F:/HTML2DRPG/combat-io.js)

需要扩充协议，至少新增：

- 时间轴快照
- 当前行动者
- 插入窗口
- 终结技资源
- 技能节奏信息

## 5.6 [view-models.js](/F:/HTML2DRPG/view-models.js)

需要新增：

- `combatTimelineViewModel`
- `ultimateViewModel`
- `combatMenuTimingViewModel`

用于把内部时间轴转换成玩家能看懂的信息。

## 5.7 [entities.js](/F:/HTML2DRPG/entities.js)

技能配置需要补齐：

- `baseDelay`
- `advanceSelf`
- `delayTarget`
- `ultimateChargeGain`
- `ultimateChargeCost`
- `actionType`
- `insertRules`

这一步先不要求数值最终正确，但字段边界要先立起来。

## 5.8 [main.js](/F:/HTML2DRPG/main.js)

需要减少“直接驱动战斗按钮行为”的逻辑，改成：

- 从时间轴视图模型读当前行动信息
- 从战斗菜单视图模型读技能列表和终结技入口
- 根据插入窗口控制终结技按钮显隐与可用状态

## 6. UI 改造建议

## 6.1 常驻区域

战斗时应新增一个小型时间轴条，展示：

- 接下来 4 到 6 个行动节点
- 当前行动者
- 己方 / 敌方区分

## 6.2 技能按钮信息

技能按钮不能只显示：

- 技能名
- MP 消耗

还需要显示：

- 延迟倾向
- 是否提前自己
- 是否延后敌人
- 是否增加终结技充能

## 6.3 终结技入口

建议不要把终结技混进普通技能列表。  
应作为单独入口，并满足：

- 平时可见但可能未充能
- 插入窗口打开时强提醒
- 释放后有明确的“插入成功”反馈

## 6.4 日志与反馈

日志中应明确写出：

- 谁推进到了前面
- 谁被延后了
- 终结技是否插入成功
- 当前行动序列是否改变

不然玩家会感受不到系统的复杂度价值。

## 7. 推荐开发顺序

这轮不要从 UI 先动。  
最佳顺序是：

1. 建立 `CombatTimelineState`
2. 改造 `CombatActionSpec`
3. 让战斗可以在无 UI 下完成时间轴推进
4. 再接时间轴 UI
5. 最后接终结技插入交互

## 8. 最低可交付版本

如果要控制正式版前的风险，建议先做 `Lite 版`：

- 只支持 `1v1`
- 只支持单敌人 AV 推进
- 只支持 1 条终结技资源槽
- 只支持少量时间轴改动效果
  - `advanceSelf`
  - `delayTarget`
  - `speedBuff`
  - `speedDebuff`

先把核心体验做对，再决定要不要继续加深。

## 9. 这轮不碰什么

这次改造先不处理：

- 职业最终平衡
- 敌人最终数值重做
- 完整多目标协同
- 复杂群体战站位
- 终结技动画大演出

否则会把系统改造和内容平衡混在一起，节奏必乱。

## 10. 验收标准

这轮技术改造完成后，至少应达到：

- `combat.js` 不再依赖 `playerTurn` 布尔值作为核心推进模型
- 时间轴可以在无 UI 情况下推进并输出快照
- 技能可以通过配置表达延迟、提前和延后
- 终结技可以通过独立协议插入当前行动序列
- UI 可以展示下一批行动者和终结技插入机会

如果这五点达不到，就说明这轮只是“换了术语”，不是“改了系统”。

## 11. V0.9 下一阶段落地建议

在 V0.7 收口后，下一阶段不要直接跳去职业平衡。
最佳顺序是先把“信息轴”补齐，再接“压制轴”。

### 11.1 R1 敌方意图与信息轴首轮接入

目标：

- 让敌方回合在真正执行前先生成 `pendingEnemyIntent`
- 让快照、视图模型、敌情面板和行动提示都读取同一份意图
- 让终结技插入窗口与敌方意图共用同一条时机提示语义

最低交付：

- `combat.js` 能在敌方回合开始时冻结敌方下一动作意图
- `combat-io.js` 能把 `enemyIntent` 写入快照
- `view-models.js` / `main.js` 能稳定展示敌方意图名称、摘要和时机提示
- `validate-combat.js` 能验证敌方意图在插入窗口前已可读

### 11.2 R2 压制轴底座（韧性 / 失衡 / 架势）

目标：

- 为敌我双方建立统一的压制资源结构
- 让打断、压制伤害、失衡收益和处决窗口有统一承接接口

最低交付：

- 单位运行时补齐 `poise` / `stagger` / `stance` 或等价字段
- 动作协议补齐压制伤害、蓄力层级和打断标记
- 快照能输出压制进度，UI 至少能在血条或敌情面板中读到

当前状态：

- 已完成。敌我双方现在都具备独立压制运行时，快照会输出韧性、失衡窗口、架势和蓄力字段
- 技能菜单会显示韧性削减，敌情面板与行动提示会在失衡窗口中切换到压制语义
- 战斗烟雾测试已覆盖“击破 -> 失衡 -> 恢复”的基础闭环

### 11.3 R3 敌方蓄力、打断与处决反馈

目标：

- 让压制轴从底层字段变成玩家真正能围绕它做决策的玩法

最低交付：

- 至少一类敌方蓄力技能会明确展示“可打断”风险
- 打断成功会改变敌方行动结果或时间轴位置
- 失衡 / 处决窗口会在日志、提示条和敌情面板中形成闭环反馈

当前状态：

- 已完成。部分敌方动作现在会先进入蓄力阶段，再在下一拍释放真正的重击
- 玩家若在蓄力阶段击穿敌方韧性，会直接打断动作并把敌方压后到后续时间轴
- 失衡中的敌方在轮到行动时会被迫踉跄恢复，无法直接按原计划完整出手
- 战斗专项验证已覆盖蓄力预告、打断成功、失衡踉跄和恢复链路

### 11.4 R4 职业技能首轮挂轴内容化

目标：

- 把 `蓄力 / 打断 / 失衡 / 处决` 从底层规则升级为职业技能的真实差异点
- 让玩家在技能菜单里就能看懂“这招该拿来打断，还是拿来等处决窗口”

验收标准：

- 至少五个职业的代表技能具备围绕蓄力或失衡窗口的专属收益
- 技能菜单能显示 `蓄力特攻`、`处决收益` 与当前是否 `可打断`
- 战斗专项验证覆盖“凭借挂轴技能打断蓄力”的实际链路

当前状态：

- 已完成。战士、法师、游侠、盗贼、圣骑士的首批核心技能现在都带有明确的挂轴字段
- 技能菜单会在平时显示蓄力特攻 / 处决收益，在敌方蓄力阶段额外给出可打断判断
- 战斗专项验证已改为使用挂轴技能完成一次真实打断，确保内容化规则不是只停留在配置层
