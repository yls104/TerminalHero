# A级文档：开发规则

## 1. 目的

这份文档用于约束后续开发方式，避免项目再次出现：

- 先堆功能后补底座
- 计划与实现脱节
- 任务完成后没有同步记录
- 版本推进过程中缺少明确的状态追踪

## 2. 需求拆解规则

每一个版本目标都必须拆成“待完成需求”。

每条需求至少要包含：

- 需求名称
- 所属版本
- 用户价值
- 依赖项
- 验收标准
- 当前状态

状态只允许使用：

- `未开始`
- `进行中`
- `已完成`
- `阻塞`

## 3. 开发路径拆解规则

每一个需求都必须再拆成“待完成步骤”。

每个步骤至少要说明：

- 目标
- 责任模块
- 输入
- 输出
- 验证方式

这样做的目的是避免只停留在“有想法，但没有抓手”的层面。

## 4. 架构优先规则

如果一个新功能会显著增加耦合度，而当前底座不足以承载，那么必须先重构底座，再开发该功能。

禁止出现以下路径：

- 明知主文件过重，仍继续往里堆系统
- 明知数据结构不足，仍直接开发遗物、技能树、复杂事件
- 明知状态边界混乱，仍继续扩大战斗和地图逻辑

## 5. 配置优先规则

新增内容时优先采用配置扩展，而不是流程代码硬编码。

优先配置化的对象包括：

- 职业
- 技能
- 敌人
- 区域
- 事件
- 掉落
- 奖励

## 6. 验证规则

每次完成开发时，至少要完成以下动作：

1. 对涉及文件做可运行验证
2. 检查是否引入同类耦合问题
3. 确认是否需要同步更新 README 文档
4. 更新进度追踪文档
5. 再进行 `git commit` 与 `git push`

## 7. 进度登记规则

这是硬规则，不是建议。

每次完成一个开发项时，必须在 [A4-进度追踪.md](/F:/HTML2DRPG/README/A4-进度追踪.md) 中同步更新：

- 对应需求状态
- 对应步骤状态
- 完成日期
- 简短结果说明

如果没有更新进度文档，则该开发项不视为真正完成。

## 8. 文档同步规则

若发生以下任一情况，必须同步更新文档：

- 产品方向发生调整
- 开发顺序发生调整
- 数据结构边界发生变化
- 新增硬性开发规则
- 进度状态发生变化

## 9. 提交规则

每轮正式更新后默认执行：

1. `git commit`
2. `git push`

只有用户明确要求“这次不要提交/不要推送”时，才可以跳过。
