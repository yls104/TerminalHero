# A级文档：开发规则

## 1. 目的

这份文档用于约束后续开发方式，避免项目再次出现：

- 先堆功能后补底座
- 计划与实现脱节
- 任务完成后没有同步记录
- 版本推进过程中缺少明确状态追踪
- 系统复杂度上升后玩家体验反而恶化

## 2. 需求拆解规则

每一个版本目标都必须拆成“待完成需求”。

每条需求至少要包含：

- 需求名称
- 所属版本
- 用户价值
- 依赖项
- 验收标准
- 当前状态

状态只允许使用：

- `未开始`
- `进行中`
- `已完成`
- `阻塞`

## 3. 开发路径拆解规则

每一个需求都必须再拆成“待完成步骤”。

每个步骤至少要说明：

- 目标
- 责任模块
- 输入
- 输出
- 验证方式

这样做的目的，是避免只停留在“有想法，但没有抓手”的层面。

## 4. 架构优先规则

如果一个新功能会显著增加耦合度，而当前底座不足以承载，那么必须先重构底座，再开发该功能。

禁止出现以下路径：

- 明知主文件过重，仍继续往里堆系统
- 明知数据结构不足，仍直接开发遗物、技能树、复杂事件
- 明知状态边界混乱，仍继续扩大战斗和地图逻辑

## 5. 配置优先规则

新增内容时优先采用配置扩展，而不是流程代码硬编码。

优先配置化的对象包括：

- 职业
- 技能
- 敌人
- 区域
- 事件
- 掉落
- 奖励

## 6. 玩家体验优先规则

这是硬规则，不是建议。

### 6.1 内部可以复杂，玩家界面必须简单直觉

允许底层系统复杂，但不允许把复杂度原样甩给玩家。

必须保证：

- 玩家主界面只看到当前决策必需的信息
- 高级机制通过渐进展示出现
- 同一概念在不同界面中的名称、图标、描述一致
- 玩家不需要理解底层数据结构就能完成决策

### 6.2 每个交付版本都必须过体验门槛

以下任一情况存在，该版本不得标记为完成：

- 主路径交互明显违背直觉
- 关键状态缺乏反馈
- 关键信息看不清、看不懂或找不到
- 移动端横屏下主视口、按钮触达、文字可读性存在重大问题
- 新系统上线后让原有信息层级更混乱

### 6.3 数据变更必须同步 UI 映射

凡是新增、删除或调整玩家关键数据字段，必须同步说明：

- 哪些信息常驻显示
- 哪些信息按场景显示
- 哪些信息收进详情面板
- 哪些信息只保留内部使用

如果未同步 UI 映射文档，则该数据变更不视为真正完成。

## 7. 验证规则

每次完成开发时，至少要完成以下动作：

1. 对涉及文件做可运行验证
2. 检查是否引入同类耦合问题
3. 确认是否需要同步更新 README 文档
4. 更新进度追踪文档
5. 再进行 `git commit` 和 `git push`

如果改动涉及核心体验，还必须补做：

1. 复核 [S1-产品大纲.md](/F:/HTML2DRPG/README/S1-产品大纲.md) 是否仍然一致
2. 复核 [A2-数据架构重构规划.md](/F:/HTML2DRPG/README/A2-数据架构重构规划.md) 的数据边界是否被破坏
3. 复核 [A5-游戏框架设计.md](/F:/HTML2DRPG/README/A5-游戏框架设计.md) 的 UI 与事件承接是否仍成立

## 8. 进度登记规则

这是硬规则，不是建议。

每次完成一个开发项时，必须在 [A4-进度追踪.md](/F:/HTML2DRPG/README/A4-进度追踪.md) 中同步更新：

- 对应需求状态
- 对应步骤状态
- 完成日期
- 简短结果说明

如果没有更新进度文档，则该开发项不视为真正完成。

## 9. 文档同步规则

若发生以下任一情况，必须同步更新文档：

- 产品方向发生调整
- 开发顺序发生调整
- 数据结构边界发生变化
- 游戏框架承接方式发生变化
- 玩家关键体验规则发生变化
- 进度状态发生变化

推荐同步顺序：

1. `S1-产品大纲`
2. `A2-数据架构重构规划`
3. `A5-游戏框架设计`
4. `A1-后续开发计划`
5. `A4-进度追踪`

## 10. 提交规则

每轮正式更新后默认执行：

1. `git commit`
2. `git push`

只有用户明确要求“这次不要提交 / 不要推送”时，才可以跳过。
