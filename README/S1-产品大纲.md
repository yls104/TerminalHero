# S级文档：产品大纲

## 1. 项目定位

- 项目名称：`Terminal Brave`
- 当前形态：原生前端实现的 2D 俯视角探索 + 回合制 RPG 原型
- 目标形态：具备稳定单轮冒险闭环、可扩展内容生产能力、支持移动端横屏稳定游玩的轻量 RPG

这个项目接下来不能再按“先堆功能，最后整理底座”的方式推进。  
后续每一步都要同时考虑：

- 玩家是否因此获得更强的选择和驱动力
- 工程结构是否还能支撑后续继续扩展

## 2. 当前已有基础

- 城镇、职业、技能、装备、战斗、关卡、Boss、回城循环已经具备雏形。
- 手机横屏版本已经有独立布局和触控方案，不再是桌面界面硬缩。
- 关卡已经从固定地图升级为程序化生成，并且开始有主题差异。
- 不同区域的小怪和 Boss 已经有基础数值区分与战斗角色区分。

这些说明项目已经从“单一玩法 demo”走到了“系统原型”阶段。

## 3. 当前最核心的不足

### 3.1 核心循环还不够强

现在玩家能做的是：

- 进图
- 清怪
- 打 Boss
- 回城

但这条链路还缺一个关键问题的答案：玩家为什么愿意再来一轮？

目前不足在于：

- 掉落期待感不强
- 回城成长的反馈还偏弱
- 刷旧图和推新图的收益差异不清晰
- 单轮冒险中的事件变化还不够多

如果不补这一层，程序化关卡只会变成“随机重复”，不会变成“可重复游玩”。

### 3.2 养成系统还停留在浅层

当前有职业、技能、装备、技能点，但离真正的 RPG 构筑还差一层：

- 技能树或职业专精
- 装备词条或装备成长
- 明确的流派分化
- 长线资源目标

现在更像“玩家每轮都变强一点”，而不是“玩家正在打造自己的 build”。

### 3.3 程序化生成还主要停留在地图层

目前随机的是：

- 地形
- 怪物点位
- 区域主题

真正成熟的程序化内容还应该继续发展到：

- 房间功能差异
- 事件节点
- 精英怪与奖励点
- 可选支线
- 风险收益选择

否则后面即使加更多素材，玩家仍然会感受到结构单一。

### 3.4 城镇系统还不像真正的 Hub

现在城镇承担了职业、商店、出征入口这些功能，但还不像一个有生命力的中心区域。

后续城镇应该承担：

- 成长反馈
- 资源消耗
- 长期目标
- 世界推进感

如果城镇只是菜单集合，那么“回城”就只是流程切换，不是情绪节点。

### 3.5 最大隐患：工程底座重构如果继续后置，会拖垮后续所有版本

这是当前最大的架构风险，不是普通的技术债提醒。

已知风险：

- `main.js` 已经承担了过多职责
- 游戏状态、UI 状态、关卡状态没有完全分层
- 战斗逻辑、状态结算、UI 渲染仍然耦合较深
- 数据定义还没有彻底从流程代码中抽离

如果在这样的底座上继续直接叠加：

- 精英怪
- 事件节点
- 技能树
- 遗物
- 复杂掉落

那么耦合度会快速上升，后面再做模块拆分和数据配置外置时，重构成本会明显高于开发新功能。

结论：
工程底座和数据结构重构必须前置到下一阶段最前面，而不能放在 V0.5 再做。

## 4. 当前数据结构是否足以支撑后续玩法

结论很明确：`不够`。

当前数据结构能支撑：

- 固定职业与技能
- 基础战斗流程
- 简单程序化地图
- 基础商店和装备

但它还不足以稳定支撑以下复杂玩法：

- 技能树 / 职业专精
- 遗物系统
- 多层掉落体系
- 事件节点与条件分支
- 多阶段关卡目标
- 配置驱动的敌人与区域生成
- 存档与读档

如果不先规划最终形态的数据边界，接下来的开发就是空中楼阁。

## 5. 最终产品形态

后续所有规划都应围绕这个目标形态展开。

### 5.1 玩家体验目标

玩家进入游戏后应形成以下稳定循环：

1. 在城镇整理职业构筑、装备和资源
2. 选择一张区域地图进入冒险
3. 在关卡中遇到普通战、精英战、事件、奖励点和回复点
4. 通过路线和资源规划决定打法
5. 清怪后开启 Boss 房
6. 击败 Boss 后回城结算本轮收益
7. 利用收益继续强化 build，挑战更高区域或回刷旧图

### 5.2 产品内容目标

最终至少应形成以下系统层：

- 城镇 Hub 系统
- 关卡生成系统
- 战斗与状态系统
- 职业与技能树系统
- 装备与遗物系统
- 掉落与奖励系统
- 区域与敌人主题系统
- 存档与进度系统

### 5.3 工程目标

最终项目不应继续以一个超重 `main.js` 作为总入口承载全部逻辑，而应具备：

- 核心状态管理层
- 战斗领域模块
- 关卡生成领域模块
- 数据配置层
- UI 渲染层
- 存档与进度层

## 6. 接下来的产品原则

### 原则一：先修底座，再扩玩法

不是所有“能让玩家更爽”的功能都应该立刻先做。  
如果底座已经明显不够，必须先解决底座。

### 原则二：每个新系统都必须增强“选择”

后续新增内容不能只是“数值更高”或“按钮更多”。

必须至少增强其中一项：

- 战斗中的选择
- 路线中的选择
- 成长中的选择
- 资源分配中的选择

### 原则三：优先建设内容生产能力

项目以后最值钱的不是“再写死一张图”，而是建立一套能持续产出内容的结构。

优先建设：

- 数据配置表
- 关卡规则模板
- 事件模板
- 敌人模板
- 奖励模板

### 原则四：移动端体验继续保持主视口优先

横屏移动端已经成为项目的重要目标平台，后续所有新 UI 都必须默认先考虑：

- 主视口占比
- 左手触达
- 操作层级
- 文案可读性
- 单轮操作负担

## 7. 距离真正完成还差什么

我把“真正完成”定义成以下 4 个标准：

1. 玩家有连续玩 3 到 5 轮的理由。
2. 开发侧新增一个职业、地图、怪物、事件时成本可控。
3. 手机横屏实机体验稳定，不需要人为兜底修正。
4. 数据结构和模块边界足以支撑继续扩展，而不是每次都返工主逻辑。

按这个标准判断，项目现在更接近“`系统原型 35% 到 45%`”，而不是“`可交付成品 80%`”。

问题不在于缺几个功能，而在于系统之间还没有形成真正的乘法关系。

## 8. 当前最值钱的下一步

最值钱的一步已经不是立刻继续加职业或事件，而是：

1. 前置工程底座拆分
2. 明确最终数据结构
3. 把配置化边界立住
4. 然后再做单轮冒险闭环增强

只有这样，后续所有玩法扩展才不会在错误地基上继续加层。
