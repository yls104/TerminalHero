# S级文档：产品大纲

## 1. 项目定位

- 项目名称：`Terminal Brave`
- 当前形态：原生前端实现的 2D 俯视角探索 + 回合制 RPG 原型
- 目标形态：具备稳定单轮冒险闭环、可持续扩展内容生产能力、支持移动端横屏稳定游玩的轻量 RPG

这个项目后续不能再按“先堆功能，最后整理底座”的方式推进。  
后续每一步都必须同时满足两件事：

- 玩家侧：增加明确选择、增强反馈、降低理解成本
- 开发侧：降低耦合、稳定扩展、避免返工

## 2. 当前已经站住的基础

- 已经具备城镇、职业、技能、装备、战斗、关卡、Boss、回城循环的雏形
- 已经开始具备程序化关卡生成和区域主题差异
- 已经为移动端横屏建立独立布局，而不是桌面页面硬缩
- 已经开始把职业差异从“数值差异”推进到“打法差异”

这些说明项目已经从“能跑的 demo”进入“系统原型”阶段。

## 3. 当前最核心的不足

### 3.1 核心循环仍然偏弱

玩家目前能完成“进图、清怪、打 Boss、回城”，但仍然缺少足够强的重复驱动力。

主要问题：

- 掉落期待感不足
- 回城成长反馈偏弱
- 刷旧图和推新图的收益差异不够清晰
- 单轮冒险中的事件变化不够多

如果不补这一层，程序化关卡只会变成“随机重复”，而不会变成“可重复游玩”。

### 3.2 养成系统还没有形成真正的构筑感

当前已有职业、技能、装备、技能点，但离真正的 RPG 构筑还差一层：

- 技能树或职业专精
- 装备词条或装备成长
- 明确的流派分化
- 长线资源目标
- 可直接查阅的装备 / 遗物 / 技能详细数值
- 面向不同职业资源池的统一 UI 承接方式

现在更像“每轮稍微变强”，还不是“玩家在打造自己的 build”。

### 3.3 程序化生成目前主要停留在地图层

当前随机的是：

- 地形
- 怪物点位
- 区域主题

真正成熟的程序化内容还应该发展到：

- 房间功能差异
- 事件节点
- 精英怪与奖励点
- 可选分支
- 风险收益选择

否则后续即使继续加素材，玩家仍然会感到结构单一。

### 3.4 城镇系统还不像真正的 Hub

城镇已经承担了职业、商店、出征入口等功能，但还不像一个有生命力的中心区域。

后续城镇必须承担：

- 成长反馈
- 资源消耗
- 长期目标
- 世界推进感

如果城镇只是菜单集合，那么“回城”就只是流程切换，而不是情绪节点。

### 3.5 最大隐患仍然是工程底座

这是当前最大的架构风险，不只是一般意义上的技术债。

已知风险：

- `main.js` 仍然承担过多职责
- 游戏状态、UI 状态、关卡状态没有完全分层
- 战斗逻辑、状态结算、UI 渲染仍然存在直接耦合
- 数据定义还没有彻底脱离流程代码

如果在这样的底座上继续叠加精英怪、事件、技能树、遗物和复杂掉落，后续重构成本会远高于新功能本身。

结论：工程底座和数据结构重构必须前置，而不是后置。

## 4. 当前数据结构是否足以支撑后续玩法

结论很明确：`不够`。

当前数据结构能支撑：

- 固定职业与技能
- 基础战斗流程
- 简单程序化地图
- 基础商店与装备

但还不足以稳定支撑：

- 技能树 / 职业专精
- 遗物系统
- 多层掉落体系
- 事件节点与条件分支
- 多阶段关卡目标
- 配置驱动的敌人与区域生成
- 存档与读档

如果不提前规划最终形态的数据边界，接下来的开发就是空中楼阁。

## 5. 最终产品形态

### 5.1 玩家体验目标

玩家进入游戏后应形成如下稳定循环：

1. 在城镇整理职业构筑、装备和资源
2. 选择区域地图进入冒险
3. 在关卡中遭遇普通战、精英战、事件、奖励点和恢复点
4. 通过路线与资源规划决定打法
5. 清理小怪后解锁 Boss 房
6. 击败 Boss 后回城结算本轮收益
7. 利用收益继续强化 build，挑战更高区域或回刷旧图
8. 在章节线完成后进入终局挑战模式，持续刷新层数和积分纪录

### 5.2 产品内容目标

最终至少应形成以下系统层：

- 城镇 Hub 系统
- 关卡生成系统
- 战斗与状态系统
- 职业与技能树系统
- 装备与遗物系统
- 掉落与奖励系统
- 区域与敌人主题系统
- 终局冲层与长期纪录系统
- 存档与进度系统

### 5.3 工程目标

项目最终不应继续依赖一个超重的 `main.js` 承载全部逻辑，而应形成：

- 核心状态管理层
- 战斗领域模块
- 关卡生成领域模块
- 数据配置层
- UI 展示层
- 存档与进度层

## 6. 必须统一遵守的产品原则

### 原则一：先修底座，再扩玩法

只要底座明显不够，就必须先解决底座问题，而不是继续堆功能。

### 原则二：每个新系统都必须增强“选择”

新增内容不能只是“数值更高”或“按钮更多”，至少要增强以下之一：

- 战斗中的选择
- 路线中的选择
- 成长中的选择
- 资源分配中的选择

### 原则三：内部可以复杂，玩家侧必须直觉

系统内部允许复杂，但玩家界面不能把复杂度原样暴露出去。

必须坚持：

- 玩家只看到完成决策所需的信息
- 同一个概念在不同界面中的名称和表现要一致
- 高级机制通过渐进展示出现，而不是一开始全堆给玩家
- 任何影响胜负的重要状态，都必须有明确反馈
- 装备、遗物、技能如果会影响构筑决策，就必须提供可直接查阅的详细数值与机制说明
- 职业专属资源如果会影响构筑或战斗决策，就必须有统一的 HUD / 场景提示 / 详情说明方案

### 原则四：数据结构不能直接等于 UI 结构

数据层负责正确表达规则与状态，UI 层负责把这些状态转换成玩家能读懂的视图。

不允许出现：

- 直接把内部字段名当成玩家文案
- 把所有运行时字段堆进面板里
- 让玩家为了理解系统去“读数据结构”

### 原则五：移动端体验继续保持主视口优先

移动端横屏已经成为重要目标平台，后续新 UI 必须默认先考虑：

- 主视口占比
- 左手触达
- 操作层级
- 文案可读性
- 单轮操作负担

### 原则六：每个交付版本都不能带着重大体验问题通过

版本完成不仅是“功能跑通”，还必须满足：

- 不存在会误导玩家的核心 UI
- 不存在关键状态缺少反馈的情况
- 不存在主路径上明显违背直觉的交互
- 不存在输入、阅读、移动端布局上的重大障碍

只要上述问题存在，该版本就不能视为真正完成。

## 7. 距离真正完成还差什么

我把“真正完成”定义为以下 4 个标准：

1. 玩家有连续玩 3 到 5 轮的理由
2. 开发侧新增职业、地图、怪物、事件时成本可控
3. 手机横屏实机体验稳定，不需要人工兜底修正
4. 数据结构和模块边界足以支撑继续扩展，而不是每次返工主逻辑

按这个标准判断，项目现在更接近“系统原型 35% 到 45%”，而不是“可交付成品 80%”。

## 8. 当前最值钱的下一步

当前最值钱的一步不是继续堆职业或事件，而是：

1. 前置工程底座拆分
2. 明确最终数据结构
3. 建立数据到 UI 的稳定映射规则
4. 再基于这套底座强化单轮冒险闭环

只有这样，后续所有玩法扩展才不会建立在错误地基上。
