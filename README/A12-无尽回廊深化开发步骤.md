# A级文档：无尽回廊深化开发步骤

## 1. 文档目的

这份文档用于把 `V1.3-R2 回廊专属敌人机制与词缀深化` 从“方向判断”拆成真正可执行的开发步骤。  
目标不是继续给无尽回廊堆系统数量，而是让它从“可反复冲层的数值模式”升级为“有独立战斗身份的终局玩法”。

## 2. 本轮深化目标

本轮要解决的核心问题有三个：

1. 高层目前主要依赖数值抬升，缺少终局模式自己的机制辨识度
2. 玩家进入高层前，无法明确预判这一层的特殊压力和规则考题
3. 无尽回廊还没有形成一套可持续扩展的“词缀 -> 敌人 -> UI -> 奖励”挂接结构

因此本轮深化的总目标是：

- 建立回廊专属词缀系统
- 让普通层、精英层、首领层围绕词缀形成不同战斗考题
- 让玩家在进层前、战斗中、结算后都能读懂这层为什么危险
- 为后续 `V1.3-R3 专属奖励池` 预留稳定挂点

## 3. 需求拆解

### V1.3-R2-1 回廊词缀配置底座

用户价值：
让无尽回廊的高层不只是“怪更肉更疼”，而是每层有明确的规则变化和构筑压力。

验收标准：

- 词缀以配置形式定义，而不是散写在主流程里
- 词缀可描述作用对象、触发时机、效果说明和 UI 文案
- 楼层生成时能稳定挂接 1 个或多个词缀

### V1.3-R2-2 普通层 / 精英层 / 首领层差异化机制

用户价值：
让玩家在不同层类型中面对不同的战斗决策，而不是只是换一批更强的敌人。

验收标准：

- 普通层以基础压力词缀为主
- 精英层会引入更强的节奏干扰或压制加速词缀
- 首领层会叠加独立的高压机制或双阶段词缀

### V1.3-R2-3 高层预警与战斗内反馈

用户价值：
让玩家在进入一层前就能知道这层危险点在哪，而不是靠挨打一遍学习规则。

验收标准：

- 守门人或进层提示中可展示当前层词缀摘要
- 战斗 HUD 中可展示当前层核心词缀
- 结算界面可回顾本轮主要词缀压力

### V1.3-R2-4 验证与难度标定

用户价值：
确保新增机制不会破坏现有战斗系统，同时无尽回廊的失败曲线仍然稳定成立。

验收标准：

- 运行时校验覆盖词缀生成与可读性
- 战斗专项校验覆盖词缀接线不破坏时间轴 / 压制轴 / 结算链路
- 文档中保留一份层数与词缀的难度标定规则

## 4. 开发步骤

### V1.3-R2-S1 设计词缀协议与数据结构

目标：
先把词缀定义为稳定的数据协议，而不是先写临时特判。

责任模块：
- `stage-data.js`
- `README/A2-数据架构重构规划.md`
- `README/A11-无尽回廊计划.md`

输入：
- 现有无尽回廊楼层结构
- 现有战斗字段：时间轴、韧性、失衡、蓄力、终结插入

输出：

- `corridorAffix` 配置结构
- 楼层到词缀的分配规则
- 作用对象、作用阶段、说明文案和 UI 标签定义

验证方式：

- 文档中明确配置结构
- 运行时可生成带词缀元数据的楼层快照

### V1.3-R2-S2 先落地 3 组基础词缀原型

目标：
用最小可玩的原型验证这套词缀系统真的能改变战斗体验。

建议首批词缀：

- `抢轴压迫`
  敌方获得更强的时间轴推进或开局优势
- `韧性再生`
  敌方在特定回合恢复部分韧性，惩罚拖节奏打法
- `处决禁区`
  首领层在特定窗口外降低处决收益，迫使玩家重新判断爆发时机

责任模块：
- `stage-data.js`
- `combat.js`
- `combat-effects.js`
- `view-models.js`

输入：
- 词缀协议
- 当前敌人与 Boss 机制

输出：

- 至少 3 个可运行的词缀原型
- 可被楼层快照读取的词缀说明

验证方式：

- 手动测试能感知到玩法差异
- 战斗专项验证不报错

### V1.3-R2-S3 把词缀挂到普通层 / 精英层 / 首领层生成规则

目标：
让楼层类型真正决定规则压力，而不是只影响数值倍率。

责任模块：
- `stage-data.js`
- `main.js`

输入：
- 楼层类型规则
- 词缀原型

输出：

- 普通层词缀池
- 精英层词缀池
- 首领层专属词缀或词缀组合规则

验证方式：

- 多次生成楼层时，能稳定看到不同层类型拿到不同词缀
- 首领层词缀不会和普通层完全同质化

### V1.3-R2-S4 增加进层预警和战斗内承接

目标：
让词缀从“底层规则”变成“玩家看得懂的战斗信息”。

责任模块：
- `main.js`
- `view-models.js`
- `style.css`
- `index.html`

输入：
- 楼层词缀快照

输出：

- 守门人入口可读的高层简报
- 进入某层时的词缀预警
- 战斗 HUD 中的词缀摘要
- 结算时的本轮压力回顾

验证方式：

- 玩家无需打开详情面板也能知道这层为什么危险
- 词缀文案不会与职业机制、区域压力文案冲突

### V1.3-R2-S5 难度标定与验证补齐

目标：
确保词缀深化后，无尽回廊仍保持“最终必然失败，但失败原因可读”的终局曲线。

责任模块：
- `scripts/validate-runtime.js`
- `scripts/validate-combat.js`
- `README/A11-无尽回廊计划.md`

输入：
- 词缀系统
- 楼层生成规则
- 战斗快照结构

输出：

- 词缀生成与展示校验
- 词缀不破坏 AV / 压制 / 结算链路的专项验证
- 难度标定说明

验证方式：

- 默认验证脚本通过
- 文档中明确记录“词缀是如何抬升失败曲线”的原则

## 5. 推荐开发顺序

严格按以下顺序推进：

1. `S1` 词缀协议
2. `S2` 三个原型词缀
3. `S3` 分层挂接
4. `S4` UI 预警与战斗承接
5. `S5` 校验与难度标定

原因：

- 如果没有统一协议，后面一定会回到主流程堆特判
- 如果先做 UI 不先做词缀原型，展示层会变成空壳
- 如果不在最后补验证，回廊会很快变成“每加一个词缀就怕炸系统”的状态

## 6. 本轮不做的内容

为了避免边界失控，这一轮先不做：

- 回廊专属奖励池本体
- 赛季制轮换主题
- 专属排行榜
- 多房间路线分支
- 回廊专属事件链

这些内容统一保留给下一阶段 `V1.3-R3`。
