# A级文档：数据架构重构规划

## 1. 目标

当前项目的数据结构只能支撑“原型期玩法”，不足以安全支撑后续的：

- 技能树
- 遗物系统
- 复杂掉落
- 事件节点
- 多阶段关卡
- 存档系统

本规划的目标，是提前定义项目最终需要的数据结构边界，避免后续开发建立在脆弱底座上。

## 2. 当前结构的不足

### 2.1 玩家数据过于扁平

当前玩家数据多为单层字段，适合基础战斗，不适合复杂成长。

不足表现：

- 缺少职业成长分支信息
- 缺少临时状态和永久状态的清晰边界
- 缺少遗物、装备词条、资源包等分层结构
- 不利于存档和版本迁移

### 2.2 技能定义还不够抽象

当前技能结构适合少量固定技能，但对以下场景支撑不足：

- 技能树解锁关系
- 技能升级
- 多目标 / 条件触发
- 状态类技能的统一结算

### 2.3 敌人与关卡配置不够模板化

当前区域和敌人虽然开始有主题，但仍偏“逻辑内嵌 + 局部配置”。

不足表现：

- 区域规则和地图生成规则耦合
- 敌人的掉落、事件权重、生成权重缺少统一结构
- 精英怪和 Boss 的差异还未被独立建模

### 2.4 缺少真正的运行态与配置态分离

现在很多数据既承担“静态配置”的角色，又承担“运行时状态”的角色。

这会导致：

- 配置难复用
- 状态难存档
- 测试难隔离
- 功能扩展时容易互相污染

## 3. 最终应具备的数据层次

## 3.1 配置层

只描述“是什么”，不描述“当前变成了什么”。

应包含：

- 职业配置
- 技能配置
- 装备配置
- 遗物配置
- 敌人模板配置
- 区域模板配置
- 事件模板配置
- 掉落模板配置

其中装备配置与遗物配置还必须具备：

- 明确的数值加成结构
- 面向玩家的详细说明结构
- 可用于构筑详情面板的 inspect 数据

## 3.2 运行态层

只描述“本轮游戏当前是什么状态”。

应包含：

- 玩家当前属性
- 当前关卡状态
- 当前事件状态
- 当前战斗状态
- 当前临时增益 / 减益
- 当前资源与背包

## 3.3 持久化层

只描述“哪些内容需要跨轮或跨会话保留”。

应包含：

- 解锁职业 / 区域
- 玩家长期资源
- 已拥有装备 / 收藏
- 当前存档位信息
- 版本号与迁移字段

当前长期资源建议落在 `progress.longTerm` 中，至少包含：

- `legacyMarks`
  可消费的长期货币，用于城镇建设与永久投资
- `townRenown`
  不可逆累积的城镇沉淀，用于后续章节和世界推进
- `townUpgrades`
  已购买的永久升级项目等级，如生命上限、法力上限、收益加成

章节推进建议落在 `progress.chapterProgress` 中，至少包含：

- `unlockedChapterIds`
  当前已经开放给玩家挑战的章节编号
- `clearedStageIds`
  已完成首领收口的区域，用于章节状态与守门人展示
- `currentChapterId`
  当前世界推进所处章节，用于章节文案和阶段引导
- `campaignComplete`
  是否已经完成当前版本的整条章节线

## 3.4 展示层视图模型

这是本轮新增必须明确的一层。  
它不属于配置层，也不属于核心运行态层，而是负责把内部数据转换成玩家可理解的信息。

应包含：

- 常驻 HUD 视图数据
- 战斗菜单视图数据
- 角色详情面板视图数据
- 区域信息视图数据
- 奖励结算视图数据
- 构筑详情视图数据
- 职业资源池视图数据

原则：`内部状态 != 玩家看到的结构`。

## 4. 建议的核心数据模型

## 4.1 PlayerState

应拆成：

- `identity`
- `baseStats`
- `derivedStats`
- `resources`
- `build`
- `inventory`
- `runtimeStatus`
- `progress`

为了承接职业体验差异，`PlayerState` 还必须支持：

- `classArchetype`
  表示职业的输出模型与风险模型，例如爆发型、连击型、持续压制型、循环型
- `combatIdentity`
  表示职业在战斗中的核心行为特征，例如先手爆发、状态铺垫、资源转伤害、护盾转输出
- `resourceLoops`
  表示该职业依赖的资源循环，不局限于 MP，还应支持怒气、连击点、印记、姿态、过载值等
- `burstWindows`
  记录职业的强势输出窗口，用于承接“准备期 -> 爆发期 -> 回落期”的体验
- `recoveryTools`
  记录职业在失误或资源见底时如何重建节奏
- `classResource`
  统一描述当前职业专属资源的标识、名称、当前值、上限、展示样式和说明文案
- `buildSnapshot`
  统一描述当前装备、遗物、关键被动、技能分支和可供玩家查看的构筑摘要

## 4.2 SkillConfig

应包含：

- 基础信息
- 消耗
- 目标类型
- 结算类型
- 状态效果
- 解锁条件
- 升级关系
- 标签

为了支撑职业体验，还应额外具备：

- `comboTags`
- `consumes`
- `generates`
- `windowRules`
- `fallbackRole`
- `inspect`
  面向玩家展示的详细数值、条件、收益和说明文案

## 4.3 EnemyTemplate

应包含：

- 基础数值
- 行为角色
- 技能池
- 掉落池
- 生成权重
- 所属区域
- 是否精英 / 是否 Boss

为了支撑职业差异，还必须具备：

- `pressureProfile`
  描述敌人如何施压，例如抢节奏、拖回合、堆护甲、爆发斩杀
- `counterTags`
  描述该敌人天然克制或检验哪些职业机制

## 4.4 StageTemplate

应包含：

- 主题信息
- 美术资源键
- 地图生成规则
- 怪物池
- 精英池
- 事件池
- 奖励池
- Boss 房规则

为了放大职业体验差异，区域模板还必须具备：

- `tempoBias`
  描述该区域鼓励爆发、消耗、拉扯还是持久战
- `classPressureTags`
  描述该区域重点考验哪些职业体验

## 4.5 RunState

这是单轮冒险的核心运行态，建议单独存在。

应包含：

- 当前区域
- 当前章节或层级
- 当前地图实例
- 本轮获得的遗物
- 当前路线事件结果
- 当前奖励记录

为了支撑职业体验，还应包含：

- `classRuntime`
  记录本轮与职业机制直接相关的运行时数据，例如连击点、誓言层数、印记层数、姿态值
- `combatMemory`
  记录上一场战斗延续下来的职业状态

## 4.6 CombatTimelineState

如果正式版前引入行动值时间轴系统，战斗运行态还必须新增独立的时间轴模型。

至少包含：

- `actors`
  当前参战单位的运行态快照
- `currentActorId`
  当前获得行动权的单位
- `queuePreview`
  供 UI 展示的后续行动序列
- `pendingInsertWindow`
  当前是否存在终结技插入窗口
- `roundIndex`
  当前大轮次，用于日志、回放和状态期限结算
- `lastResolvedAction`
  最近一次结算完成的动作摘要

## 4.7 SkillTimingConfig

若技能进入时间轴系统，技能配置必须补齐节奏字段。

至少包含：

- `baseDelay`
  该技能的基础行动延迟
- `advanceSelf`
  该技能是否让自己提前下次行动
- `delayTarget`
  该技能是否让目标延后行动
- `speedScale`
  速度是否会影响该技能的延迟收益
- `ultimateChargeGain`
  使用后获得多少终结技充能
- `ultimateChargeCost`
  若为终结技，释放时消耗多少终结技资源
- `insertRules`
  是否允许作为终结技插入，以及可插入的窗口类型

## 5. 数据到 UI 的显示映射规则

这是后续所有数据结构设计必须遵守的硬规则。

### 5.1 玩家不应直接阅读内部字段

内部字段服务于规则结算，UI 服务于玩家决策。  
因此不允许：

- 直接把字段名暴露给玩家
- 直接把 `runtimeStatus` 原样堆进面板
- 让玩家通过“看一堆状态字段”理解玩法

### 5.2 玩家可见信息必须分层

玩家可见信息固定分为四层：

- `常驻层`
  只显示生命、法力、经验、等级、核心职业资源等即时决策必需信息
- `上下文层`
  只在相关场景出现，例如战斗窗口、区域压力、Boss 机制、技能可用条件
- `详情层`
  通过“详细属性”“技能详情”“区域情报”等入口查看
- `隐藏层`
  仅内部使用，不直接给玩家展示

### 5.3 同一概念必须保持单一命名

例如：

- 数据层叫“连击点”，UI 就不能一会儿叫“连击”，一会儿叫“节奏值”
- 数据层叫“爆发窗口”，UI 应统一为对应职业能理解的同一概念

### 5.4 UI 应基于视图模型，而不是直接读领域对象

推荐每个核心界面都有对应视图模型：

- `hudViewModel`
- `combatMenuViewModel`
- `characterPanelViewModel`
- `stageInfoViewModel`
- `rewardSummaryViewModel`
- `buildCodexViewModel`
- `classResourceViewModel`
- `combatTimelineViewModel`
- `ultimateInsertViewModel`
- `combatMenuTimingViewModel`

领域对象负责正确，视图模型负责可读。

### 5.5 复杂机制必须渐进揭示

新增复杂机制时，UI 表达规则如下：

- 常驻层只显示对当前决策有直接影响的结果
- 关键状态提供简短提示
- 详细解释放入详情面板或技能说明
- 不把未来阶段才重要的信息提前堆给玩家

### 5.6 构筑信息必须可检视

凡是会影响玩家构筑决策的对象，必须满足以下之一：

- 在详情层可查看完整数值
- 在战斗场景中可查看关键成本与收益
- 在主路径 UI 中至少能知道“为什么值得选”

不允许出现：

- 装备只有名字，没有具体属性
- 遗物只有稀有度和标签，没有实际收益说明
- 技能只有名字和描述，没有成本、倍率、生成 / 消耗关系

## 6. 模块边界建议

## 6.1 Config 模块

负责所有静态配置读取与组合。

## 6.2 Domain 模块

负责纯规则与结算逻辑。

建议至少拆出：

- `combat`
- `run`
- `progression`
- `stage_generation`
- `class_runtime`
- `effect_pipeline`
- `reward_resolution`

## 6.3 UI 模块

只处理显示，不直接承载战斗公式和成长逻辑。  
同时负责从领域数据生成视图模型。

## 6.4 Persistence 模块

负责存档、读档、版本迁移。

当前建议采用 `SaveSnapshotV1` 作为最小稳定存档结构，至少包含：

- `player`
  当前玩家可持久化状态，如职业、属性、技能、装备、遗物、材料、金币、技能点
- `progress`
  长线推进状态，如已解锁区域、已击破首领
  同时包含长期资源、章节推进与世界解锁状态
- `currentStage`
  当前所处区域、关卡模式、地图快照、节点池、当前位置
- `runSummary`
  当前单轮冒险中已经累计的收益摘要
- `merchantStock`
  当前城镇商人的在售装备实例

不应写入存档的内容包括：

- 纯 UI 开关状态
- 临时弹层显示状态
- 当前帧动画与插值状态
- 依赖当前回调栈的战斗中间态

## 7. 近期重构优先级

### 第一优先级

- 将职业、技能、敌人、区域定义从流程逻辑中抽离成配置对象
- 区分静态配置和运行时状态
- 为职业专属资源循环建立可扩展字段
- 建立玩家可见信息的视图模型层

### 第二优先级

- 抽离战斗结算核心
- 抽离关卡生成核心
- 建立效果结算管线
- 建立视图模型生成规则与 UI 绑定协议

### 第三优先级

- 为存档留出稳定字段
- 为遗物、事件、掉落预留扩展位
- 为版本迁移预留数据修正入口

## 8. 验收标准

完成本轮数据结构重构后，应达到：

- 新增一个敌人模板主要通过配置完成
- 新增一个区域模板不需要频繁修改主流程代码
- 战斗逻辑可不依赖 UI 单独描述输入输出
- 玩家状态可明确区分长期进度、单轮进度和临时状态
- UI 能通过视图模型稳定展示玩家可理解的数据
- 新增一个职业不需要修改战斗主循环本身，只需补充配置、效果定义和少量框架挂点

如果达不到这些点，说明重构还没有真正完成。
