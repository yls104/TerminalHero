# A级文档：职业设计方案

## 1. 设计前提

这是一个单人 RPG 冒险游戏，不是多人协作 MMO。

因此职业设计的前提必须改成：

- 所有职业本质上都是输出职业
- 不存在只负责团队分工的纯坦克或纯治疗
- 防御、恢复、控制存在的意义，是帮助玩家更高效地输出并活到下一轮

职业差异不在于“谁能不能打”，而在于：

- 怎么打
- 什么时候强
- 失误后怎么把节奏拉回来

## 2. 职业体验设计原则

每个职业都必须具备四个维度：

1. 核心输出模型
2. 风险与容错模型
3. 资源循环模型
4. 爆发窗口模型

如果一个职业只剩技能名字不同、倍率不同，那它就不是独立职业，只是换皮。

## 3. 七职业差异体验

## 3.1 战士

### 核心体验

`压上去打穿`

### 体验特征

- 输出建立在近身压迫和短时爆发上
- 不是“很肉”，而是“能顶着压力把伤害打满”
- 玩家感受到的是正面压制感

### 玩家决策

- 什么时候开启爆发
- 什么时候先上增益再重击
- 什么时候为下一个窗口忍一手

### 数据结构支持

- 怒气 / 压制值
- 爆发窗口标记
- 高风险高回报技能标签

### 框架支持

- 支持先铺垫再引爆
- 支持防守换输出窗口

### 当前重构落地补充

- 起手压制：`裂风斩`
- 窗口启动：`战吼`
- 失衡处决：`裂地猛击`
- 回稳换压：`不屈战吼`

战士已经开始从“高血高攻”改成“围绕压制窗口运作”的新版模板职业。

## 3.2 法师

### 核心体验

`资源换爆发`

### 体验特征

- 输出峰值高，但对资源和时机要求高
- 玩家感受到的是“把资源精确砸在正确窗口里”

### 玩家决策

- 什么时候用高消耗法术
- 什么时候回蓝
- 什么时候用控制拉出施法空间

### 数据结构支持

- 法力循环状态
- 过载 / 咏唱 / 元素层数等扩展字段

### 框架支持

- 支持资源恢复技能
- 支持技能前后状态联动

## 3.3 游侠

### 核心体验

`拉扯与持续压制`

### 体验特征

- 不是瞬时最高爆发，而是持续把压力压在敌人身上
- 玩家感受到的是“每一回合都在赚节奏”

### 玩家决策

- 什么时候挂持续伤害
- 什么时候精准射击
- 什么时候进入长线消耗而不是强行爆发

### 数据结构支持

- 标记状态
- 持续伤害来源记录
- 目标弱点标签

### 框架支持

- 支持持续伤害独立结算
- 支持标记对后续技能加成

### 当前重构落地补充

- 起手拖轴：`瞄准射击`
- 持续压制：`毒箭`
- 拉扯回稳：`应急包扎`
- 收割处决：`箭雨`

游侠已开始从“稳定远程输出”改成“围绕拖轴、压血和远程收割运作”的新版模板职业。

## 3.4 牧师

### 核心体验

`稳定推进后的审判输出`

### 体验特征

- 恢复和护持不是终点，而是为了把审判伤害打出来
- 玩家感受到的是“稳住局面，然后惩戒收尾”

### 玩家决策

- 什么时候转恢复
- 什么时候转惩戒
- 什么时候先保护自己为下回合输出做准备

### 数据结构支持

- 祝福 / 审判印记
- 回复转输出条件标签

### 框架支持

- 支持护盾 / 恢复与伤害链路互动
- 支持“准备期 -> 惩戒期”的转换

## 3.5 盗贼

### 核心体验

`抢节奏与斩杀`

### 体验特征

- 强调先手、连击、斩杀窗口
- 玩家感受到的是“抓住一瞬间打出致命伤害”

### 玩家决策

- 什么时候积累连击点
- 什么时候直接斩杀
- 什么时候用闪避保住下一个爆发

### 数据结构支持

- 连击点
- 先手加成
- 斩杀阈值

### 框架支持

- 支持连击资源生成与消耗
- 支持目标血线阈值触发效果

### 当前重构落地补充

- 起手抢轴：`背刺`
- 连段续压：`淬毒刃`
- 回合修复：`烟幕步`
- 斩杀处决：`影袭乱舞`

盗贼已开始从“高速高爆发”改成“围绕抢轴、修回合和斩杀阈值运作”的新版模板职业。

## 3.6 圣骑士

### 核心体验

`防守反击与稳态爆发`

### 体验特征

- 不是纯抗，而是通过护盾和稳态推进为自己创造爆发
- 玩家感受到的是“我不一定最快，但我会越来越稳，最后压死你”

### 玩家决策

- 什么时候转防守
- 什么时候用圣光回稳
- 什么时候把安全局面转成终结爆发

### 数据结构支持

- 护盾值
- 反击窗口
- 神圣充能

### 框架支持

- 支持护盾与伤害转换
- 支持防守行为触发后续输出收益

## 3.7 德鲁伊

### 核心体验

`状态循环与自然节奏`

### 体验特征

- 擅长维持持续恢复、持续压制和状态轮转
- 玩家感受到的是“整场战斗都被我慢慢掌控”

### 玩家决策

- 什么时候挂恢复
- 什么时候补护甲
- 什么时候把铺好的状态转成集中收益

### 数据结构支持

- 自然印记 / 生长层数
- 持续恢复状态
- 周期性触发计数

### 框架支持

- 支持持续效果多层并行
- 支持状态转化和引爆

## 4. 七职业的核心差异总结

| 职业 | 核心体验 | 输出方式 | 风险特征 |
| --- | --- | --- | --- |
| 战士 | 压上去打穿 | 爆发压制 | 贴脸吃压 |
| 法师 | 资源换爆发 | 高耗高伤 | 断蓝后脆弱 |
| 游侠 | 拉扯与压制 | 持续消耗 | 爆发上限较晚出现 |
| 牧师 | 稳定推进后审判 | 回复转输出 | 伤害启动偏慢 |
| 盗贼 | 抢节奏与斩杀 | 先手连击 | 失误后容易掉节奏 |
| 圣骑士 | 防守反击 | 稳态爆发 | 爆发依赖积累 |
| 德鲁伊 | 状态循环 | 持续控制场面 | 收益需要时间堆叠 |

## 5. 数据结构必须提供的支持

为了让这七职业真正成立，数据层必须至少支持：

- 通用资源字段 + 职业专属资源容器
- 技能生成 / 消耗资源的定义
- 爆发窗口与状态窗口定义
- 目标阈值与标记系统
- 持续效果与状态转化系统
- 容错恢复手段的显式建模

## 5.1 未来开发目标：围绕共享战斗维度丰富职业体验

正式版前，职业体验不应只依赖“职业专属资源”这一条线。  
还应围绕一组共享战斗维度生长出不同的职业与遗物流派。

这些维度先作为未来开发目标冻结如下：

- `速度`
  未来不只决定先手，还要决定上轴速度、短延迟循环、连续行动、终结技插入时机
- `防御`
  未来不只减少伤害，还应支撑反击、受击充能、以守换攻、硬吃换轴这类玩法
- `生命 / 已损生命`
  未来不只提供容错，还应支撑残血爆发、失血换收益、过量治疗转护盾或输出
- `法力`
  未来不只用于支付技能费用，还应支撑过载、资源换节奏、空蓝爆发等玩法
- `攻击`
  未来不只决定伤害倍率，还应与重击、破防、处决阈值、压制能力挂钩
- `终结能量`
  未来应成为独立于法力和职业资源之外的第三类共享战斗资源，用于支撑插入型爆发体验
- `韧性 / 失衡 / 架势`
  未来建议新增为共享战斗维度，用于支撑打断、压制、处决与反压制玩法

这些共享维度的价值在于：

- 让职业差异不只长在“按钮不同”
- 让遗物和装备可以围绕共享战斗语言生长构筑
- 让未来职业平衡更多是挂接到统一系统，而不是继续堆特判

## 5.2 未来开发目标：七职业与共享维度的关系

后续正式开发时，建议优先让七职业围绕以下共享维度长出差异：

- 战士
  偏 `防御 / 韧性 / 重击 / 爆发窗口`
- 法师
  偏 `法力 / 终结能量 / 慢轴高收益`
- 游侠
  偏 `速度 / 持续压制 / 延后敌人行动`
- 牧师
  偏 `恢复转收益 / 稳态推进 / 终结插入收尾`
- 盗贼
  偏 `速度 / 连续行动 / 斩杀阈值`
- 圣骑士
  偏 `防御 / 护盾 / 反击 / 稳态爆发`
- 德鲁伊
  偏 `状态循环 / 韧性消磨 / 节奏拉扯`

这部分先定义为未来开发目标，不在当前版本承担数值平衡任务。

### 当前落地进展补充

- 已完成首轮“职业技能挂轴”接入：战士、法师、游侠、盗贼、圣骑士的代表技能开始围绕 `蓄力目标` 与 `失衡目标` 产生差异收益
- 本轮落地优先承接五类直觉体验：`打断蓄力`、`压低韧性`、`延后敌方行动`、`顺着失衡窗口处决`、`借爆发窗口抢轴收尾`
- 后续职业深化与专精扩展，应继续沿用这组共享战斗语言扩张，而不是回到单纯“高倍率技能”的旧设计

## 6. 玩家界面必须如何承接职业体验

职业体验只有写进 UI，玩家才真正能感受到。

### 6.1 常驻显示

常驻层只显示：

- 生命值
- 法力值
- 等级
- 经验值
- 当前职业的核心资源

不允许把所有职业机制常驻堆满 HUD。

### 6.2 场景显示

按场景显示：

- 当前爆发窗口是否开启
- 当前关键状态是否满足技能条件
- 当前敌人或区域是否正在施加特定压力

这些信息应在战斗或区域相关场景里抬升，而不是永久常驻。

### 6.3 详情显示

详情层负责解释：

- 职业核心机制
- 资源循环说明
- 爆发窗口说明
- 关键技能联动
- 装备、遗物、技能的具体数值
- 当前职业资源的生成 / 消耗规则与上限

详情层是“帮助理解”，不是“代替主界面做决策”。

### 6.5 不同职业资源池的统一承接规则

七个职业虽然资源类型不同，但 UI 承接必须统一满足：

- HUD 中始终有同一位置展示当前职业核心资源
- 战斗菜单中能看见技能对该资源的生成与消耗
- 详情层中能查到该资源的完整规则说明

不能出现某个职业靠常驻 HUD 展示，另一个职业却只能靠日志或技能描述猜测。

### 6.4 命名一致性

职业资源、职业窗口、职业印记在所有界面中的名称必须一致。  
不能出现数据层一种叫法，HUD 一种叫法，技能说明又一种叫法。

## 7. 游戏框架必须提供的支持

为了让职业体验真正落地，框架必须至少支持：

- 效果管线
- 资源事件
- 状态事件
- 窗口事件
- 技能条件校验
- 运行态容器
- UI 显示挂点
- 视图模型生成

没有这些，职业差异最终会退化成一堆特殊判定。

## 8. 对现有七职业的理解修正

后续正式开发时，不建议再把职业理解成：

- 战士更肉
- 牧师更能奶
- 圣骑士更稳

而应理解成：

- 战士：压制型爆发输出
- 法师：资源型爆发输出
- 游侠：持续压制型输出
- 牧师：稳定推进型输出
- 盗贼：斩杀型输出
- 圣骑士：防守反击型输出
- 德鲁伊：循环型输出

这才符合单人 RPG 的职业设计逻辑。

## 9. 后续开发含义

这份职业设计不是写给文案单独看的，而是直接决定：

- 数据结构怎么改
- 战斗系统怎么拆
- UI 应该显示什么
- 技能系统需要预留什么扩展位

如果这些内容没有同步到数据结构和框架层，职业设计仍然只是纸面方案。
