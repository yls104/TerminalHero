# A级文档：职业设计方案

## 1. 设计前提

这是一个单人 RPG 冒险游戏，不是多人协作 MMO。

因此职业设计的前提必须改成：

- 所有职业本质上都是输出职业
- 没有“纯坦克”或“纯治疗”这种只负责团队分工的职业
- 防御、回复、控制存在的意义，是为了帮助玩家更高效地输出和活到下一轮

也就是说，职业差异不在于“谁能不能打”，而在于“怎么打”“什么时候强”“失误后怎么把节奏拉回来”。

## 2. 职业体验设计原则

每个职业都必须具备以下 4 个维度：

1. 核心输出模型
2. 风险与容错模型
3. 资源循环模型
4. 爆发窗口模型

如果一职业只剩下技能名不同、倍率不同，那它就不是独立职业，只是换皮。

## 3. 七职业差异体验

## 3.1 战士

### 核心体验

`压上去打穿`

### 体验特征

- 输出建立在近身压迫和短时爆发上
- 不是“很肉”，而是“能顶着压力把伤害打满”
- 玩家感受到的是正面压制感

### 玩家决策

- 什么时候开爆发
- 什么时候先上增益再重击
- 什么时候为了下一个窗口忍一手

### 数据结构支持

- 怒气/压制值
- 爆发窗口标记
- 高风险高回报技能标签

### 框架支持

- 支持先铺垫再引爆
- 支持防御换输出窗口

## 3.2 法师

### 核心体验

`资源换爆发`

### 体验特征

- 输出峰值高，但对资源和时机要求高
- 玩家感受到的是“把资源精确砸在正确窗口里”

### 玩家决策

- 什么时候用高消耗法术
- 什么时候回蓝
- 什么时候通过减速或控制拉出施法空间

### 数据结构支持

- 法力循环状态
- 过载/咏唱/元素层数等扩展字段

### 框架支持

- 支持资源恢复技能
- 支持技能前后状态联动

## 3.3 游侠

### 核心体验

`拉扯与持续压制`

### 体验特征

- 不是瞬时最高爆发，而是持续把压力压在敌人身上
- 玩家感受到的是“每一回合都在赚节奏”

### 玩家决策

- 什么时候挂毒
- 什么时候精准射击
- 什么时候进入持续消耗而不是强行爆发

### 数据结构支持

- 标记状态
- 持续伤害来源记录
- 目标弱点标签

### 框架支持

- 支持持续伤害独立结算
- 支持标记对后续技能加成

## 3.4 牧师

### 核心体验

`稳定推进后的审判输出`

### 体验特征

- 回复和护持不是终点，而是为了保证自己能把审判伤害打出来
- 玩家感受到的是“稳住局面，然后惩戒收尾”

### 玩家决策

- 什么时候转恢复
- 什么时候转惩戒
- 什么时候利用保护技能为下回合输出做准备

### 数据结构支持

- 祝福/审判印记
- 回复转输出的条件标签

### 框架支持

- 支持护盾/回复与伤害链路互动
- 支持“准备期 -> 惩戒期”的转换

## 3.5 盗贼

### 核心体验

`抢节奏与斩杀`

### 体验特征

- 强调先手、连击、斩杀窗口
- 玩家感受到的是“抓住一瞬间打出致命伤害”

### 玩家决策

- 什么时候积累连击点
- 什么时候直接斩杀
- 什么时候用闪避保住下一轮爆发

### 数据结构支持

- 连击点
- 先手加成
- 斩杀阈值

### 框架支持

- 支持连击资源生成与消费
- 支持目标血线阈值触发效果

## 3.6 圣骑士

### 核心体验

`防守反击与稳态爆发`

### 体验特征

- 不是纯抗，而是通过护盾和稳态推进为自己创造爆发
- 玩家感受到的是“我不一定最快，但我会越来越稳，最后压死你”

### 玩家决策

- 什么时候转防守
- 什么时候用圣光回稳
- 什么时候把安全局面转成终结爆发

### 数据结构支持

- 护盾值
- 反击窗口
- 神圣充能

### 框架支持

- 支持护盾与伤害转换
- 支持防守行为触发后续输出收益

## 3.7 德鲁伊

### 核心体验

`状态循环与自然节奏`

### 体验特征

- 擅长维持持续恢复、持续压制和状态轮转
- 玩家感受到的是“不是一击定胜负，而是整场战斗都被我慢慢掌控”

### 玩家决策

- 什么时候挂恢复
- 什么时候补护甲
- 什么时候把铺好的状态转成集中收益

### 数据结构支持

- 自然印记/生长层数
- 持续恢复状态
- 周期性触发计数

### 框架支持

- 支持持续效果多层并行
- 支持状态转化和引爆

## 4. 七职业的核心差异总结

| 职业 | 核心体验 | 输出方式 | 风险特征 |
| --- | --- | --- | --- |
| 战士 | 压上去打穿 | 爆发压制 | 贴脸吃压 |
| 法师 | 资源换爆发 | 高耗高伤 | 一旦断蓝就弱 |
| 游侠 | 拉扯与压制 | 持续消耗 | 爆发上限较晚出现 |
| 牧师 | 稳定推进后审判 | 回复转输出 | 伤害启动偏慢 |
| 盗贼 | 抢节奏与斩杀 | 先手连击 | 失误后容易掉节奏 |
| 圣骑士 | 防守反击 | 稳态爆发 | 爆发期依赖积累 |
| 德鲁伊 | 状态循环 | 持续控制场面 | 收益需要时间堆叠 |

## 5. 数据结构必须提供的支持

为了让这七职业真的成立，数据层必须至少支持：

- 通用资源字段 + 职业专属资源容器
- 技能生成/消耗资源的定义
- 爆发窗口与状态窗口的定义
- 目标阈值与标记系统
- 持续效果与状态转化系统
- 容错恢复手段的显式建模

## 6. 游戏框架必须提供的支持

为了让职业体验真正落地，框架必须至少支持：

- 效果管线
- 资源事件
- 状态事件
- 窗口事件
- 技能条件校验
- 运行态容器
- UI 显示挂点

没有这些，职业差异最终会退化成一堆特殊判定。

## 7. 对现有七职业的调整建议

后续正式开发时，不建议再把职业理解成：

- 战士更肉
- 牧师更能奶
- 圣骑士更稳

而应该理解成：

- 战士：压制型爆发输出
- 法师：资源型爆发输出
- 游侠：持续压制型输出
- 牧师：稳定推进型输出
- 盗贼：斩杀型输出
- 圣骑士：防守反击型输出
- 德鲁伊：循环型输出

这样才符合单人 RPG 的职业设计逻辑。

## 8. 后续开发含义

这份职业设计不是写给美术或文案看的，而是直接决定后续：

- 数据结构怎么改
- 战斗系统怎么拆
- UI 应该显示什么
- 技能系统需要预留什么扩展位

如果这些内容没有同步到数据结构和框架层，职业设计就仍然是纸面方案。
