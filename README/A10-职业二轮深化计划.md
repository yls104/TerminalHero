# A级文档：职业二轮深化计划

## 1. 为什么在 V1.1 之后必须进入二轮深化

V1.1 的目标，是把七职业从旧回合制语义中拉出来，统一接到新的战斗语言上。
这一步已经完成了三个关键收口：

- 七职业都已经恢复可选
- 每个职业都具备了可读的资源循环、技能职责与专精方向
- HUD、战斗菜单、构筑详情与专项校验已经能承接“时间轴 / 压制轴 / 处决窗口”这套共通语言

但这还不等于职业已经真正“长成”。
当前版本仍然存在三个明显缺口：

1. 职业差异更多停留在“技能职责不同”，还没有稳定落到“专属机制如何驱动决策”
2. UI 虽然能读懂现有资源与职责，但还不能稳定承接“职业机制当前处于什么阶段、何时值得兑现”
3. 缺少一套共享的职业二轮协议，导致后续若直接给每个职业硬塞专属判定，代码和体验都会再次漂移

所以，V1.2 的目标不是继续“补更多技能”，而是让七职业正式进入：

`共享协议 -> UI 承接 -> 职业专属机制 -> 专项验证`

这条更稳定的深化路径。

## 2. V1.2 的总目标

V1.2 要完成的不是数值平衡，而是职业机制层的正式立项和第一批落地：

- 为七职业建立统一的二轮职业协议
- 让玩家能在详情面板、构筑手册和技能检视里读懂“职业机制现在是什么状态”
- 先以战士作为第一批二轮深化样板，把“共享协议”真正跑通
- 为后续盗贼、游侠、法师、圣骑士、德鲁伊、牧师的二轮深化预留稳定接线方式

V1.2 的核心原则：

1. 先共享，后专属
   没有共享协议前，不允许继续堆新的职业特判。
2. 先可读，后复杂
   任何新机制都必须先能被玩家读懂，才允许继续往上叠层。
3. 先样板，后扩散
   先做一个真正成立的战士二轮样板，再用同一协议迁移其它职业。

## 3. 共享职业二轮协议

从 V1.2 起，每个职业都必须同时具备以下四层信息：

### 3.1 静态职业机制档案

用于定义这个职业在二轮深化里的核心身份：

- `mechanicName`
  这个职业当前最重要的二轮机制名称
- `mechanicSummary`
  一句话说明它到底在围绕什么做决策
- `battleLoop`
  这个职业一轮战斗中最典型的三段式或多段式循环
- `decisionAxes`
  玩家在战斗中最常面对的关键取舍
- `uiSignals`
  这个职业机制应该在哪些界面里被承接

### 3.2 运行时职业机制状态

用于定义机制在战斗中的当前阶段：

- 当前值
- 上限
- 是否进入可兑现状态
- 当前状态文案
- 当前阶段适合做什么

### 3.3 技能与职业机制的挂接关系

每个职业都必须显式说明：

- 哪些技能负责“铺机制”
- 哪些技能负责“稳住机制”
- 哪些技能负责“兑现机制”
- 当机制进入准备完成状态后，哪些技能会发生强化

### 3.4 UI 承接规范

职业机制的信息必须分层展示：

- 常驻层
  详细属性面板中要能看到职业机制名称、当前状态和当前读数
- 详情层
  构筑手册里要能完整解释循环、收益、风险和决策点
- 上下文层
  技能检视与技能按钮要能在必要时提示“当前技能是否已进入强化状态”

## 4. V1.2 的开发拆解

### V1.2-R1 共享职业协议与 UI 承接

目标：
把二轮深化需要的职业档案、运行时机制状态、构筑手册承接与详情面板承接全部接上。

交付：

- 七职业补齐 `secondPassProfile`
- 玩家运行时补齐统一的职业机制状态容器
- 详细属性面板新增职业机制读数与状态
- 构筑手册新增“职业机制”承接区块
- 运行时校验覆盖职业档案、详情面板与构筑手册输出

验收标准：

- 玩家不看代码，也能在 UI 里知道当前职业二轮机制叫什么、处于什么阶段
- 七职业都具备统一字段，不再靠职业特判散落在多个文件中

### V1.2-R2 战士二轮深化样板

目标：
把战士从“只有职责语义”推进到“真的围绕专属机制做节奏决策”。

本轮定义：

- 机制名：`破军势`
- 机制定位：围绕起手压制与守势过渡积累势层，在成势回合把处决技能推入更高收益窗口

第一批落地规则：

- `裂风斩 / 战吼 / 防御姿态 / 不屈战吼` 负责积势
- 战士达到成势阈值后，`裂地猛击 / 崩山断` 进入强化状态
- 强化后的处决技能会在伤害、压制或抢轴中至少有一项获得清晰提升
- 兑现后会清空 `破军势`，重新进入下一轮积势

验收标准：

- 玩家能够明显感知“铺势 -> 成势 -> 兑现 -> 回补”的战士节奏
- 技能检视和构筑手册都能读出成势收益，而不是只在底层数值里悄悄生效

### V1.2-R3 其它职业按顺序深化

建议顺序：

1. 盗贼
2. 游侠
3. 法师
4. 圣骑士
5. 德鲁伊
6. 牧师

原因：

- 先做更依赖时间轴与抢轴表达的职业
- 再做更依赖稳态经营与状态转换的职业
- 避免在共享协议尚未验证充分前，把恢复型、转化型职业先做复杂

## 5. 七职业的二轮深化目标锚点

这些锚点先冻结为 V1.2 之后的持续开发方向，避免后续漂移：

- 战士：`破军势`
  围绕积势、成势与重击兑现
- 盗贼：`夺拍窗口`
  围绕抢轴、连段与斩杀加速
- 游侠：`猎线`
  围绕拖轴、压血与远程收割
- 法师：`过载点火`
  围绕铺垫、回蓝与高耗兑现
- 圣骑士：`誓能裁决`
  围绕稳态、守反与厚重收尾
- 德鲁伊：`生长轮转`
  围绕铺场、转化与周期爆发
- 牧师：`审判回响`
  围绕回稳、蓄印与惩戒兑现

## 6. 本阶段不做什么

为了避免二轮深化再次失焦，V1.2 当前明确不做：

- 不在这一轮重做全部职业的数值平衡
- 不在没有共享协议前继续往各职业塞更多专属按钮
- 不把尚未通过 UI 承接的新机制直接塞进战斗主流程
- 不把“文档设想”直接等同于“已上线效果”

## 7. 当前执行结论

本轮应严格按以下顺序推进：

1. 先落文档与进度追踪
2. 再接共享职业协议与 UI 承接
3. 然后实现战士二轮样板 `破军势`
4. 最后补齐专项验证并登记进度

后续职业不得跳过这条顺序直接硬做。
