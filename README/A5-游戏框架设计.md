# A级文档：游戏框架设计

## 1. 目的

这份文档不是泛泛谈“模块拆分”，而是专门回答一个问题：

为了让七个职业在单人 RPG 里带来真正不同的输出体验，框架层必须提供哪些能力？

如果框架层不能承接职业体验，那么职业差异最终只会退化成：

- 技能名字不同
- 数值倍率不同
- 颜色特效不同

这不是真正的职业设计。

## 2. 框架必须承接的职业体验

单人 RPG 中所有职业本质上都要能完成伤害输出，但输出方式不同。

框架必须支持以下差异：

- 爆发窗口型
- 持续压制型
- 先手刺杀型
- 资源循环型
- 状态铺垫型
- 防守反击型
- 节奏拉扯型

也就是说，框架不能只支持“按按钮 -> 掉血”。

## 3. 必须具备的框架层

## 3.1 Core State Layer

负责全局游戏状态切换：

- 城镇态
- 探索态
- 事件态
- 战斗态
- 结算态

职业系统不能直接写死在 UI 或地图逻辑里，而应通过状态层统一管理进入和退出时机。

## 3.2 Combat Domain Layer

负责战斗核心规则，不直接操作界面。

应至少拆分：

- 行动输入解释
- 效果结算
- 状态变化
- 回合推进
- 胜负判定

## 3.3 Effect Pipeline Layer

这是职业体验差异最关键的一层。

应支持以下顺序：

1. 输入技能
2. 校验条件
3. 生成效果
4. 排队结算
5. 触发衍生效果
6. 更新资源与状态
7. 输出结果事件

只有这样，职业才能安全实现：

- 连击
- 标记
- 资源生成与消耗
- 护盾转伤害
- 中毒/灼烧/迟缓等状态联动

## 3.4 Class Runtime Layer

负责保存职业专属运行时数据。

示例：

- 盗贼的连击点
- 圣职职业的审判印记
- 法师的过载或法力循环层数
- 战士的怒气或压制值

这层不能继续直接塞进通用玩家对象的散乱字段中。

## 3.5 UI Presentation Layer

负责把职业状态以玩家能理解的方式展示出来。

应包括：

- 当前资源条
- 职业窗口提示
- 技能可用条件
- 爆发状态反馈
- 关键状态层数展示

## 4. 战斗事件管线建议

为了满足职业差异，战斗框架建议统一发出以下事件：

- `turn_started`
- `turn_ended`
- `action_selected`
- `action_resolved`
- `resource_gained`
- `resource_spent`
- `status_applied`
- `status_expired`
- `window_opened`
- `window_closed`
- `damage_dealt`
- `damage_taken`
- `kill_confirmed`

这样做的价值是：

- UI 不需要直接读取内部计算细节
- 新职业可以通过挂接事件扩展机制
- 测试和日志更容易落地

## 5. 技能执行框架建议

技能不应只是一段倍率公式，而应走统一协议。

建议结构：

1. `cost check`
2. `requirement check`
3. `resource mutate`
4. `effect build`
5. `effect resolve`
6. `post-hook`

这样职业差异才能通过配置和效果组合实现，而不是不断往 `if/else` 里塞特殊逻辑。

## 6. 对职业体验最关键的框架支持

### 6.1 资源循环支持

框架必须允许职业拥有不同资源机制，而不是全部只依赖 MP。

### 6.2 爆发窗口支持

框架必须支持“某些职业不是每回合都强，而是在某个窗口特别强”。

### 6.3 状态联动支持

框架必须支持：

- 状态叠加
- 状态消费
- 状态转化
- 状态引爆

### 6.4 容错恢复支持

单人 RPG 的职业设计不能要求玩家一失误就完全崩盘。

框架要支持“重建节奏”的技能和机制。

## 7. 当前框架重构优先级

### 第一优先级

- 把战斗逻辑从 UI 反馈中剥离
- 抽出统一的效果结算管线

### 第二优先级

- 为职业运行态建立单独容器
- 为技能条件和资源变化建立统一协议

### 第三优先级

- 为事件节点、遗物、装备词条接入同一效果系统

## 8. 验收标准

框架层如果重构完成，应该达到：

- 新增一个职业不需要重写战斗主循环
- 新增一个职业机制不需要修改多处 UI 条件分支
- 技能、状态、资源、窗口都能通过统一协议表达
- 单人 RPG 中七职业能在“输出方式”上形成真正差异
