# A级文档：游戏框架设计

## 1. 目的

这份文档不是泛泛谈“模块拆分”，而是专门回答一个问题：

为了让七个职业在单人 RPG 里带来真正不同的输出体验，框架层必须提供哪些能力？

如果框架层不能承接职业体验，那么职业差异最终只会退化成：

- 技能名字不同
- 数值倍率不同
- 颜色特效不同

这不是真正的职业设计。

## 2. 框架必须承接的职业体验

单人 RPG 中所有职业本质上都要完成伤害输出，但输出方式不同。  
框架必须支持以下差异：

- 爆发窗口型
- 持续压制型
- 先手斩杀型
- 资源循环型
- 状态铺垫型
- 防守反击型
- 节奏拉扯型

也就是说，框架不能只支持“按按钮 -> 掉血”。

## 3. 必须具备的框架层

## 3.1 Core State Layer

负责全局游戏状态切换：

- 城镇态
- 探索态
- 事件态
- 战斗态
- 结算态

职业系统不能直接写死在 UI 或地图逻辑里，而应通过状态层统一管理进入和退出时机。

## 3.2 Combat Domain Layer

负责战斗核心规则，不直接操作界面。

应至少拆分：

- 行动输入解释
- 效果结算
- 状态变化
- 回合推进
- 胜负判定

如果正式版前升级为行动值系统，这一层还必须额外支持：

- 时间轴推进
- 行动顺序预览
- 插入窗口管理
- 技能延迟与提前 / 延后结算
- 终结技独立协议

## 3.3 Effect Pipeline Layer

这是职业体验差异最关键的一层。

应支持以下顺序：

1. 输入技能
2. 校验条件
3. 生成效果
4. 排队结算
5. 触发衍生效果
6. 更新资源与状态
7. 输出结果事件

只有这样，职业才能安全实现：

- 连击
- 标记
- 资源生成与消耗
- 护盾转伤害
- 中毒 / 燃烧 / 迟缓等状态联动

## 3.4 Class Runtime Layer

负责保存职业专属运行时数据。

例如：

- 盗贼的连击点
- 牧师的审判印记
- 法师的过载或法力循环层数
- 战士的怒气或压制值

这层不能继续直接塞进通用玩家对象的散乱字段中。

## 3.5 UI Presentation Layer

负责把职业状态以玩家能理解的方式展示出来，而不是把内部结构原样扔给玩家。

必须包含：

- 常驻 HUD
- 战斗菜单
- 详情面板
- 区域信息
- 结算信息
- 构筑详情面板
- 职业资源池展示层

同时必须通过视图模型把领域数据转成玩家可读信息。

## 4. UI 展示层必须遵守的框架规则

### 4.1 视图模型是必要层，不是可选层

UI 不应直接读取领域对象。  
必须通过专门的视图模型输出：

- `hudViewModel`
- `combatMenuViewModel`
- `characterPanelViewModel`
- `stageInfoViewModel`
- `rewardSummaryViewModel`

### 4.2 信息展示必须分层

框架要支持四层展示：

- `常驻层`
  生命、法力、经验、等级、核心职业资源
- `上下文层`
  技能可用条件、Boss 机制、区域压力、战斗窗口
- `详情层`
  详细属性、职业机制解释、技能说明、敌人情报
- `隐藏层`
  只给内部规则和调试使用

### 4.3 一个概念只能有一套玩家表达

框架要保证：

- 同一概念在 HUD、详情面板、战斗提示中的文案一致
- 同一机制的图标、名称、颜色语义保持一致
- 不允许 UI 层各自发明新的叫法

### 4.4 复杂机制必须渐进揭示

框架要允许：

- 先显示结果，再允许查看解释
- 先显示当前最重要的状态，再展开机制细节
- 只在相关场景抬升复杂信息

不允许把所有可用字段一股脑堆进一个面板。

### 4.5 构筑信息必须可直接查阅

框架必须允许玩家直接查看：

- 技能的成本、倍率、关键效果和资源联动
- 装备的明确数值、词条和构筑定位
- 遗物的明确收益和触发条件
- 职业资源池的当前值、上限、生成方式和消耗方式

如果这些信息需要玩家自行推测、翻日志或读代码，说明框架没有真正承接构筑系统。

## 5. 战斗事件管线建议

为了满足职业差异，战斗框架建议统一发出以下事件：

- `turn_started`
- `turn_ended`
- `action_selected`
- `action_resolved`
- `resource_gained`
- `resource_spent`
- `status_applied`
- `status_expired`
- `window_opened`
- `window_closed`
- `damage_dealt`
- `damage_taken`
- `kill_confirmed`

如果引入时间轴与终结技插入，还应新增：

- `timeline_advanced`
- `turn_order_changed`
- `insert_window_opened`
- `insert_window_closed`
- `ultimate_ready`
- `ultimate_inserted`
- `action_delayed`
- `action_advanced`

这样做的价值是：

- UI 不需要直接读取内部计算细节
- 新职业可以通过挂接事件扩展机制
- 测试和日志更容易落地
- 构筑详情和资源池 UI 可以通过统一事件更新，不必写大量职业特判

## 6. 技能执行框架建议

技能不应只是一段倍率公式，而应走统一协议：

1. `cost check`
2. `requirement check`
3. `resource mutate`
4. `effect build`
5. `effect resolve`
6. `post-hook`

这样职业差异才能通过配置和效果组合实现，而不是不断往 `if / else` 里堆特殊逻辑。

如果接入 AV 系统，技能协议还应继续扩展：

7. `timeline mutate`
8. `insert window emit`

也就是说，技能不只影响数值，还要能影响行动序列本身。

## 7. 对职业体验最关键的框架支持

### 7.1 资源循环支持

框架必须允许职业拥有不同资源机制，而不是全部只依赖 MP。

### 7.2 爆发窗口支持

框架必须支持“某些职业不是每回合都强，而是在某个窗口特别强”。

### 7.3 状态联动支持

框架必须支持：

- 状态叠加
- 状态消耗
- 状态转化
- 状态引爆

### 7.4 容错恢复支持

单人 RPG 的职业设计不能要求玩家一失误就完全崩盘。  
框架要支持“重建节奏”的技能和机制。

### 7.5 UI 可读性支持

框架必须允许 UI 明确表达：

- 当前是否进入职业强势窗口
- 当前资源够不够用
- 技能为什么可用 / 不可用
- 当前区域或敌人正在施加什么压力
- 当前装备、遗物和技能为什么构成了当前 build
- 当前职业资源池如何生成、如何消耗、下一步是否值得进入爆发窗口

如果引入时间轴系统，UI 还必须能表达：

- 当前行动者是谁
- 接下来若干个行动者是谁
- 技能会不会拖慢自己
- 技能会不会延后敌人
- 终结技现在是否能插入
- 插入后大致会改变什么顺序

如果玩家看不懂，框架就没有真正承接职业体验。

## 8. 当前框架重构优先级

### 第一优先级

- 把战斗逻辑从 UI 反馈中剥离
- 抽出统一的效果结算管线
- 建立视图模型生成层

### 第二优先级

- 为职业运行态建立单独容器
- 为技能条件和资源变化建立统一协议
- 为区域压力与奖励倾向建立 UI 输出接口

### 第三优先级

- 为事件节点、遗物、装备词条接入同一效果系统
- 为角色详情、区域情报、结算面板接入统一展示协议

## 9. 验收标准

框架层如果重构完成，应达到：

- 新增一个职业不需要重写战斗主循环
- 新增一个职业机制不需要修改大量 UI 条件分支
- 技能、状态、资源、窗口都能通过统一协议表达
- UI 能在不暴露内部字段的前提下解释关键状态
- 单人 RPG 中七职业能在“输出方式”上形成真实差异

如果做不到这些，就说明框架还没有真正支撑玩法成长。
