# A级文档：后续开发计划

## 1. 路线总判断

原计划把“模块拆分”和“数据配置外置”放在末期，这是错误顺序。  
新的最佳路线必须改成：

1. 先重构工程底座与数据结构
2. 再做单轮冒险闭环
3. 然后升级构筑系统
4. 再深化程序化内容
5. 最后做存档、长期留存和产品化收口

如果顺序不改，后面的功能开发会持续加重耦合，最终重构成本会远超功能开发成本。

## 2. 新版本拆解

## V0.2 工程底座前置重构

### 目标

在继续扩玩法前，把当前最危险的架构问题先解决到可扩展状态。

### 核心任务

- 拆分 `main.js` 的系统职责
- 梳理核心状态机
- 将战斗公式、状态结算、UI 更新彻底解耦
- 明确地图生成、区域数据、敌人数据、职业数据的配置边界
- 建立进度追踪与开发规则执行链路

### 具体交付

- 最少拆出：`状态层`、`战斗层`、`关卡层`、`UI层`
- 关卡、敌人、职业、技能、装备的数据定义开始向配置化迁移
- 战斗结算不再直接依赖界面组件状态
- 生成规则具备模板化入口

### 验收标准

- 新增一个敌人或区域不需要频繁修改主流程文件
- 战斗系统可以不依赖 UI 独立描述输入输出
- 状态机职责边界清楚，探索态/战斗态/城镇态不再混杂

## V0.3 单轮冒险闭环

### 目标

让玩家完整体验一次“进图 -> 冒险 -> 奖励 -> Boss -> 回城”的有价值循环。

### 重点内容

- 精英怪
- 事件节点
- 战后奖励选择
- 关卡结算面板
- 回城收益反馈

### 具体交付

- 普通怪、精英怪、Boss 三层敌人结构
- 每张图至少 2 到 3 类事件节点
- 战斗结算支持金币、经验、材料、遗物中的至少两类奖励
- 回城后显示本轮收益摘要

### 验收标准

- 玩家打一轮不会只感受到“清怪重复”
- 同一区域重复进入也会出现不同节奏
- Boss 前的准备感明显增强

## V0.4 构筑系统升级

### 目标

让职业差异从“属性差异”升级为“打法差异”。

### 重点内容

- 技能树或职业专精
- 遗物系统
- 装备词条或装备品级
- 更明确的资源规划

### 具体交付

- 每个职业至少 2 条可见成长方向
- 遗物为单轮冒险提供临时 build 变化
- 装备不再只是固定加值，而是具备选择意义

### 验收标准

- 玩家会因为职业选择不同而改变打法
- 玩家会因为奖励不同而改变路线
- 回城成长不再只是线性加数值

## V0.5 程序化内容深化

### 目标

让随机生成真正服务于内容变化，而不是只服务于地图变化。

### 重点内容

- 房间/区域功能模板
- 事件池
- 精英池
- 奖励池
- 主题区域专属规则

### 具体交付

- 每个区域拥有独立的关卡生成规则
- 每个区域有独立事件池和敌人池
- 不同主题区域在路径、压力、奖励倾向上有明确区别

### 验收标准

- 玩家能明显感受到区域玩法区别
- 随机生成不再只是地形换形状
- 内容扩展可以通过配置持续加入

## V0.6 产品化与长期留存

### 目标

让项目从“能继续做”进入“能持续做”的状态。

### 重点内容

- 存档系统
- 长线资源系统
- 世界推进与区域解锁
- 基础验证链路
- 移动端最终收口

### 具体交付

- 玩家支持存档/读档
- 玩家有长期目标和中断恢复能力
- 核心流程具备最基本的验证脚本或稳定检查步骤
- 移动端体验进入稳定维护期

### 验收标准

- 后续继续扩内容不必频繁返工核心结构
- 玩家中断后可回到进度
- 项目具备继续扩大内容规模的可能性

## 3. 当前最建议直接开做的事项

不是先堆精英怪，也不是先堆遗物。

当前最优先的 4 件事是：

1. 核心状态机拆分
2. 数据结构重构与配置边界建立
3. 战斗结算与 UI 渲染解耦
4. 进度追踪与执行规则接入

只有这四件事完成后，后面的精英怪、事件、遗物、技能树才值得继续加。

## 4. 每阶段都要遵守的开发约束

- 新系统必须优先增强玩家选择，不要只增加按钮数量。
- 新 UI 必须先过移动端横屏视角，不允许回到桌面优先思路。
- 新内容尽量数据化，不要继续把内容硬写进主逻辑。
- 每轮迭代必须做最基本的可运行验证，再提交和推送。
- 每次完成一个开发项，必须同步更新进度追踪文档中的对应模块状态。
