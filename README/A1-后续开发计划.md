# A级文档：后续开发计划

## 1. 路线总判断

原计划把“模块拆分”和“数据配置外置”放在较后阶段，这是错误顺序。  
新的最佳路线必须改成：

1. 先重构工程底座与数据结构
2. 再做单轮冒险闭环
3. 然后升级构筑系统
4. 再深化程序化内容
5. 最后做存档、长期留存和产品化收口

如果顺序不改，后续功能开发只会持续加重耦合，最终重构成本会远超功能开发成本。

## 2. 版本拆解

## V0.2 工程底座前置重构

### 目标

在继续扩玩法前，先把当前最危险的架构问题解决到可扩展状态。

### 核心任务

- 拆分 `main.js` 的系统职责
- 梳理核心状态机
- 将战斗公式、状态结算、UI 更新彻底解耦
- 明确地图生成、区域数据、敌人数据、职业数据的配置边界
- 建立数据层到 UI 层的显示映射规则
- 建立进度追踪与开发规则执行链路

### 具体交付

- 至少拆出：`状态层`、`战斗层`、`关卡层`、`UI层`
- 关卡、敌人、职业、技能、装备的数据定义开始向配置化迁移
- 战斗结算不再直接依赖界面组件状态
- UI 通过视图模型读取玩家可见数据，而不是直接读内部运行时字段
- 生成规则具备模板化入口

### 验收标准

- 新增一个敌人或区域，不需要频繁修改主流程文件
- 战斗系统可以不依赖 UI 独立描述输入输出
- 状态机职责边界清晰，探索态、战斗态、城镇态不再混杂
- 玩家面板不需要直接暴露底层数据结构就能表达关键状态

## V0.3 单轮冒险闭环

### 目标

让玩家完整体验一次“进图 -> 冒险 -> 奖励 -> Boss -> 回城”的有价值循环。

### 重点内容

- 精英怪
- 事件节点
- 战后奖励选择
- 关卡结算面板
- 回城收益反馈

### 具体交付

- 普通怪、精英怪、Boss 三层敌人结构
- 每张图至少 2 到 3 类事件节点
- 战斗结算支持金币、经验、材料、遗物中的至少两类奖励
- 回城后显示本轮收益摘要

### 验收标准

- 玩家打一轮不会只感受到“清怪重复”
- 同一区域重复进入也会出现不同节奏
- Boss 前的准备感明显增强
- 奖励和结算界面不会让玩家为“看懂系统”付出额外负担

## V0.4 构筑系统升级

### 目标

让职业差异从“属性差异”升级为“打法差异”。

### 重点内容

- 技能树或职业专精
- 遗物系统
- 装备词条或装备品级
- 更明确的资源规划
- 装备 / 遗物 / 技能的详细构筑信息展示
- 多职业资源池的统一 UI 承接

### 具体交付

- 每个职业至少 2 条可见成长方向
- 遗物为单轮冒险提供临时 build 变化
- 装备不再只是固定加值，而是具备选择意义
- 重要职业机制在 UI 中可读、可理解、可追踪
- 装备、遗物、技能必须能查看具体数值、成本、收益和关键机制说明
- 每个职业的专属资源都必须定义常驻显示、上下文显示和详情显示规则

### 验收标准

- 玩家会因职业选择不同而改变打法
- 玩家会因奖励不同而改变路线
- 回城成长不再只是线性加数值
- 高级机制虽然更复杂，但主界面仍然直觉可读
- 想钻研构筑的玩家不需要靠翻日志或读代码才能理解装备、遗物、技能的具体收益
- 职业资源池切换后，不会让 HUD、战斗菜单和详情面板出现概念断裂

## V0.5 程序化内容深化

### 目标

让随机生成真正服务于内容变化，而不只是服务于地图变化。

### 重点内容

- 房间 / 区域功能模板
- 事件池
- 精英池
- 奖励池
- 主题区域专属规则

### 具体交付

- 每个区域拥有独立的关卡生成规则
- 每个区域有独立事件池和敌人池
- 不同主题区域在路线、压力、奖励倾向上有明确区别

### 验收标准

- 玩家能明显感受到区域玩法区别
- 随机生成不再只是地形换形状
- 内容扩展可以通过配置持续加入
- 新区域的 UI 文案、视觉和玩法反馈保持一致

## V0.6 产品化与长期留存

### 目标

让项目从“能继续做”进入“能持续做”的状态。

### 重点内容

- 存档系统
- 长线资源系统
- 世界推进与区域解锁
- 基础验证链路
- 移动端最终收口

### 具体交付

- 玩家支持存档 / 读档
- 玩家有长期目标和中断恢复能力
- 核心流程具备基础验证脚本或稳定检查步骤
- 移动端体验进入稳定维护期

### 验收标准

- 后续继续扩内容不必频繁返工核心结构
- 玩家中断后可回到进度
- 项目具备继续扩大内容规模的可能性

## V0.7 战斗系统升级

### 目标

把当前偏传统轮流回合制的战斗，升级为更强调节奏、窗口和操作时机的系统。

### 重点内容

- 轻量行动值（AV）时间轴
- 终结技插入
- 技能延迟与节奏收益
- 时间轴 UI 与战斗反馈
- 战斗验证链路升级

### 具体交付

- 速度属性会显著影响行动顺序和节奏收益
- 技能配置开始支持延迟、提前、延后和终结技充能
- 战斗支持终结技在特定窗口插入
- UI 可展示当前行动者、后续队列和终结技机会
- 时间轴推进可通过独立验证脚本进行烟雾测试

### 验收标准

- 玩家可以明显感知速度、延迟和插队对战斗结果的影响
- 终结技不再只是大数字按钮，而是真正的时机决策
- 战斗深度提高后，UI 仍然保持直觉可读
- 新的战斗推进模型不再依赖简单的轮流行动布尔值
- 后续再深化职业和数值时，不需要再次推翻战斗主框架

### 当前状态补充

V0.7 的首轮 AV 时间轴、终结技插入、战斗菜单与验证链路已经完成收口。
后续战斗深化不再继续回填到 V0.7，而是转入 V0.9 的“信息轴 / 压制轴”阶段推进。

## V0.8 界面体验与美术首轮重构

### 目标

把当前“功能已经齐备，但界面仍偏原型面板化”的状态，推进到“主视口优先、信息层级清楚、素材具备统一视觉方向”的可持续版本。

### 重点内容

- 探索态低干扰 HUD 重构
- 战斗态信息层级重排
- 核心角色与交互节点素材替换
- 移动端横屏进一步减负
- UI 视觉语言统一

### 具体交付

- 将探索态常驻信息收拢为一组远征摘要，而不是多个等权盒子面板
- 战斗态只抬升当前真正影响决策的敌情、时间轴和指令信息
- 区域路线、压力、奖励倾向通过统一 chip 语义呈现
- 角色、敌人、NPC、传送门等高频视觉锚点替换为更接近正式版的主题素材
- 全局字体、按钮、面板和浮层改用更贴近奇幻冒险题材的视觉语言
- 横屏移动端继续保证主视口优先，不回退到“把桌面侧栏硬塞进手机”的布局思路

### 验收标准

- 玩家进入探索态时，第一眼看到的是地图和目标，而不是一列信息墙
- 玩家能在不打开详情面板的前提下读懂当前区域、压力和收益倾向
- 战斗态中高优先级信息和低优先级信息不再使用同一视觉权重
- 角色与关键交互节点素材不再呈现明显的开发占位感
- 桌面与移动端横屏都不存在因为 UI 过重导致的主视口压迫

## V0.9 战斗信息轴与压制轴深化

### 目标

让当前“能跑的 AV 战斗”升级为“能预判、能压制、能围绕窗口做中高级决策”的战斗系统。

### 重点内容

- 敌方意图与信息轴
- 韧性 / 失衡 / 架势
- 蓄力、打断与处决窗口
- 压制反馈与血条 / 日志承接
- 战斗专项验证继续扩展

### 具体交付

- 敌方回合前会先生成并冻结敌方意图，敌情面板、提示条和终结技插入提示统一读取这份信息
- 战斗单位补齐韧性、失衡、架势和蓄力等压制字段，并进入快照与视图模型
- 技能协议开始支持压制伤害、打断条件、失衡收益和处决窗口
- 血条、敌方信息、日志和行动提示能直观表达“这招危险在哪里、能不能打断、现在是否进入压制收益窗口”
- 战斗烟雾测试覆盖敌方意图、压制累计、打断成功和处决窗口等关键链路

### 验收标准

- 玩家能在出手前看懂敌方下一拍威胁，而不是只能挨打一遍后靠记忆
- 速度、终结技插入和敌方意图会形成真实联动，而不是三套互相独立的信息
- 打断、压制和处决是规则闭环，不是只在日志里出现的新名词
- 新增的战斗维度会被 UI 直观承接，不需要玩家读底层结构猜系统
- 后续职业、遗物和 Boss 机制可以自然挂到信息轴和压制轴上继续深化

### 当前状态补充

- 已完成 `敌方意图与信息轴首轮接入`
- 下一步优先推进 `压制轴底座（韧性 / 失衡 / 架势）`

## 3. 当前最建议直接开做的事项

当前不再是继续堆系统数量的时候，而是要把已经形成的系统用更合理的 UI 和美术承接起来。

当前最优先的 5 件事是：

1. 探索态 HUD 减重，把职业、区域、资源和日志入口收拢成远征摘要
2. 战斗态信息层级重排，避免时间轴、敌情、日志和指令同权竞争
3. 用统一 chip / 卡片 / 按钮语义承接区域压力、奖励倾向和构筑状态
4. 先替换最影响观感的角色与交互锚点素材，再逐步推进区域地块替换
5. 用移动端横屏视角复核整套 UI，而不是桌面改完后再压缩适配

只有先把这五件事落地，后面的区域扩展、更多事件、更多遗物和正式美术量产才有稳定承接面。

## 4. 版本体验质量门槛

每个版本都必须同时通过功能门槛和体验门槛。  
只要体验门槛未通过，该版本不能视为真正完成。

### 体验门槛

- 不允许出现关键状态缺乏反馈的问题
- 不允许出现让玩家误解规则的主路径 UI
- 不允许出现需要玩家阅读底层数据结构才能做决策的界面
- 不允许出现移动端横屏下主视口、按钮可点性、文本可读性方面的重大问题
- 不允许出现“系统变复杂了，但信息层级更混乱”的倒退

### 数据与 UI 联动门槛

- 新增或修改玩家关键数据时，必须同步说明其 UI 展示方式
- 新增职业机制时，必须说明常驻显示、上下文显示和详情显示分别是什么
- 新增区域机制时，必须说明玩家如何在界面上理解该区域压力和奖励倾向

## 5. 每阶段都要遵守的开发约束

- 新系统必须优先增强玩家选择，而不是只增加按钮数量
- 新 UI 必须先过移动端横屏视角，不允许回到桌面优先思路
- 新内容尽量数据化，不要继续把内容硬写进主逻辑
- 每轮迭代必须做最基本的可运行验证，再提交和推送
- 每次完成一个开发项，必须同步更新进度追踪文档
- 每次变更涉及核心体验时，必须复核产品大纲、数据架构和框架设计文档的一致性
