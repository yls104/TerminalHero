# A级文档：后续开发计划

## 1. 开发路线判断

最佳路线不是“继续横向堆功能”，而是按下面顺序推进：

1. 先把单轮冒险闭环做实
2. 再把养成系统升级成构筑系统
3. 然后把程序化生成从地图随机推进到内容随机
4. 最后补工程底座和长线留存能力

这条路线的核心目标是：每增加一个系统，都能直接提升可玩性或扩展性，而不是只增加复杂度。

## 2. 版本拆解

## V0.2 单轮冒险闭环

### 目标

让玩家完整体验一次“进图 -> 冒险 -> 奖励 -> Boss -> 回城”的有价值循环。

### 重点内容

- 增加精英怪
- 增加事件节点
- 增加战后奖励选择
- 增加关卡结算面板
- 增加回城后的成长反馈

### 具体交付

- 普通怪、精英怪、Boss 三层敌人结构
- 每张图至少 2 到 3 类事件节点
- 战斗结算支持金币、经验、材料、遗物中的至少两类奖励
- 回城后显示本轮收益摘要

### 验收标准

- 玩家打一轮不会只感受到“清怪重复”
- 同一区域重复进入也会出现不同节奏
- Boss 前的准备感明显增强

## V0.3 构筑系统升级

### 目标

让职业差异从“属性差异”升级为“打法差异”。

### 重点内容

- 技能树或职业专精
- 装备词条或装备品级
- 遗物/被动系统
- 更明确的资源规划

### 具体交付

- 每个职业至少 2 条可见的成长方向
- 装备从“固定购买”升级为“属性差异化选择”
- 遗物系统为单轮冒险提供临时 build 变化

### 验收标准

- 玩家会因为职业选择不同而改变打法
- 玩家会因为奖励不同而改变路线
- 回城成长不再只是线性加数值

## V0.4 程序化内容深化

### 目标

让随机生成真正服务于内容变化，而不是只服务于地图变化。

### 重点内容

- 房间/区域功能模板
- 事件池
- 精英池
- 奖励池
- 主题区域专属规则

### 具体交付

- 每个区域拥有独立的关卡生成规则
- 每个区域有独立事件池和敌人池
- 不同主题区域在路径、压力、奖励倾向上有明确区别

### 验收标准

- 玩家能明显感受到区域玩法区别
- 随机生成不再只是地形换形状
- 内容扩展可以通过配置持续加入

## V0.5 工程底座与产品化

### 目标

让项目从“能继续做”进入“能持续做”的状态。

### 重点内容

- 存档系统
- 数据配置外置
- 模块拆分
- 基础验证链路
- 移动端最终收口

### 具体交付

- 玩家支持存档/读档
- 职业、敌人、地图、事件、装备尽量配置化
- `main.js` 按系统职责拆分
- 关键流程具备最基本的自检脚本或验证步骤

### 验收标准

- 后续继续扩内容不必频繁碰核心主文件
- 玩家中断后可回到进度
- 移动端体验进入稳定维护期，而不是持续抢修期

## 3. 推荐的开发顺序

### 第一阶段

- 精英怪
- 事件节点
- 奖励结算
- 回城收益反馈

原因：
这是最直接增强单轮体验的部分，能立刻提升“再来一轮”的动力。

### 第二阶段

- 技能树/职业专精
- 遗物系统
- 装备词条化

原因：
这是把“成长”升级成“构筑”的关键转折点。

### 第三阶段

- 区域专属生成规则
- 事件池与内容池
- 主题区域强化

原因：
这是把程序化内容从“随机地图”推进到“随机体验”。

### 第四阶段

- 存档
- 模块拆分
- 数据配置化
- 最终移动端收口

原因：
这是保证项目能继续长期迭代的工程基础。

## 4. 每阶段都要遵守的开发约束

- 新系统必须优先增强玩家选择，不要只增加按钮数量。
- 新 UI 必须先过移动端横屏视角，不允许回到桌面优先思路。
- 新内容尽量数据化，不要继续把内容硬写进主逻辑。
- 每轮迭代必须做最基本的可运行验证，再提交和推送。

## 5. 下一次最建议直接开做的事项

如果现在马上进入下一轮开发，我建议直接开做下面这组：

1. 精英怪系统
2. 事件节点系统
3. 关卡结算与回城收益面板
4. 遗物雏形

这是目前投入产出比最高的一组改动。

原因很简单：

- 它们直接增强单轮可玩性
- 它们为后续 build 系统打地基
- 它们能显著拉开“普通战斗 demo”和“可重复游玩的 RPG 原型”之间的差距
